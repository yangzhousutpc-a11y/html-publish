这份笔记回答的是一个直击灵魂的问题：Cursor、Claude Code这些"会写代码的AI"，它们到底是怎么做到的？答案出人意料地简单——只需要200行Python代码，一个精心设计的System Prompt，加上三个基础工具（读文件、写文件、执行命令），你就能搭出一个能自己写贪吃蛇游戏的AI Agent。

笔记分两大部分：第一部分讲ReAct模式——Agent边想边干的"战术执行"方式，从原理到System Prompt设计到完整Python实现，手把手走一遍；第二部分讲Plan-And-Execute模式——先做计划再逐步执行的"战略指挥"方式。两种模式的对比揭示了一个核心洞察：Agent的"智能"不来自模型本身，而来自Prompt设计和循环机制。

一、什么是Agent？

一句话：Agent = LLM（大脑）+ 工具（手脚）+ 循环执行（不停干活直到完成）。

传统LLM vs Agent

传统LLM的工作方式：
用户："帮我写一个贪吃蛇游戏"
模型：输出一段代码文本（你需要自己复制粘贴到文件里）

Agent的工作方式：
用户："帮我写一个贪吃蛇游戏"
Agent：自动创建index.html → 自动写入CSS → 自动写入JS → 完成！直接打开就能玩

区别在哪？Agent能感知外部环境（读文件）和改变外部环境（写文件、执行命令），而传统LLM只能在文字世界里打转。

Cursor、Manus、Claude Code这些编程助手，本质上都是Agent——它们用LLM做决策，用工具做执行，用循环做迭代。

📐 Agent = LLM + Tool + Loop 三位一体

💡 关键认知：Agent不需要特殊的模型训练。把一个普通的LLM加上工具和循环，它就变成了Agent。关键在于System Prompt的设计。

二、ReAct模式：边想边干

一句话：ReAct = Reasoning（推理）+ Acting（行动）。模型在一个"思考→行动→观察→再思考"的循环中工作，每一步都根据上一步的结果决定下一步做什么。

四步循环

📐 ReAct 四步循环流程图

┌─────────────────────────────────────┐
│  Thought（思考）                      │
│  "我需要先创建HTML文件"              │
└───────────────┬─────────────────────┘
                ↓
┌─────────────────────────────────────┐
│  Action（行动）                      │
│  调用 write_to_file("index.html")   │
└───────────────┬─────────────────────┘
                ↓
┌─────────────────────────────────────┐
│  Observation（观察）                  │
│  "文件写入成功"                      │
└───────────────┬─────────────────────┘
                ↓
        ┌───────┴───────┐
        │  任务完成？     │
        ├─ 否 → 回到Thought
        └─ 是 → Final Answer

每一步的含义： - Thought：模型分析当前状况，决定下一步做什么 - Action：模型请求调用某个工具（模型只"说"，不"做"） - Observation：系统真正执行工具后返回结果（模型不能自己编造！） - Final Answer：当模型认为信息足够时，输出最终结果

💡 关键认知：ReAct的精髓在于"思考可见"——模型的每一步推理都写在Thought里，这让调试变得简单。如果Agent做错了，你能看到它是在哪一步"想偏了"，而不是面对一个黑盒子。

三、System Prompt六要素：Agent的"灵魂"

一句话：System Prompt是驱动ReAct循环的"剧本"——模型不是天生就会四步循环，而是被Prompt"教会"的。

视频中揭示的System Prompt由六个核心要素组成：

📐 System Prompt 六要素 — Agent 的"灵魂配方"

# 要素 内容示例
1 角色与职责 "你是一个高效的编程助手，按ReAct范式完成任务"
2 可用工具列表 read_file、write_to_file、execute_command
3 工具描述 每个工具的功能说明和参数格式
4 输出格式规范 必须使用XML标签（\<thought>、\<action>、\<final_answer>）
5 执行规则 每次只调用一个工具、禁止自己编造observation、最多循环10次
6 环境信息 当前工作目录、操作系统、可用文件列表

💡 关键认知：Agent的"智能"不来自模型本身，而来自System Prompt的设计。一个精心设计的Prompt就能让普通模型表现得像专业Agent。这就是为什么说"System Prompt是Agent的灵魂"。

📐 XML标签通信: Agent程序↔LLM的"暗号"

四、XML格式工具调用

模型和主程序之间通过XML标签通信。模型输出的格式长这样：

<thought>需要创建一个贪吃蛇游戏，我先创建HTML文件</thought>

<action>write_to_file(index.html, <!DOCTYPE html>...)</action>

<!-- 系统执行后返回 -->
<observation>文件写入成功</observation>

<thought>HTML完成了，现在写CSS样式</thought>

<action>write_to_file(style.css, body { background: #000; }...)</action>

<observation>CSS文件写入成功</observation>

<!-- ... 循环继续 ... -->

<final_answer>贪吃蛇游戏完成！生成了3个文件。</final_answer>

为什么用XML而不用JSON？ 因为LLM生成XML比JSON更稳定——不容易出现括号不匹配的问题，而且支持嵌套，方便传递代码内容。

💡 关键认知：XML格式是Agent和程序之间的"契约"——模型保证输出特定格式的XML，程序用正则表达式解析。这比让模型输出JSON更可靠，因为LLM生成嵌套括号时容易出错，而XML标签的开闭匹配更简单。

# 工具1：读文件
def read_file(path):
    with open(path, 'r') as f:
        return f.read()

# 工具2：写文件
def write_to_file(path, content):
    with open(path, 'w') as f:
        f.write(content)
    return f"成功写入{path}"

# 工具3：执行命令
def execute_command(cmd):
    result = subprocess.run(cmd, shell=True, capture_output=True, text=True)
    return result.stdout + result.stderr

八、Plan-And-Execute模式：先谋后动

一句话：Plan-And-Execute = 先做完整计划，再逐步执行，遇到意外则动态调整计划。

三个核心组件

📐 Plan-And-Execute 三组件循环架构

用户需求
    ↓
【Planner（规划者）】→ 生成完整的步骤列表
    ↓
【Executor（执行者）】→ 逐步执行每个步骤
    ↓
【Re-Planner（重规划者）】→ 根据执行结果调整计划
    ↓
完成 or 回到 Executor

与ReAct的本质区别

ReAct是"走一步看一步"，每做完一件事才想下一件。Plan-And-Execute是"先画蓝图再施工"，一开始就有完整规划。

九、两种模式对比

📐 ReAct vs Plan-And-Execute 对比图

维度 ReAct Plan-And-Execute
执行策略 走一步看一步（动态） 先规划后执行（结构化）
比喻 战术执行者（边走边看） 战略指挥官（先谋后动）
模型数量 1个LLM 3个（Planner/Executor/Re-Planner）
Token消耗 较低（2000-3000） 较高（3000-4500）
准确率 ~85% ~92%
响应速度 较快 较慢
适用场景 简单任务、快速反应 复杂任务、高准确率
代表产品 Cursor、Claude Code Manus

💡 关键认知：两种模式不是非此即彼。实际的Agent系统往往是混合使用——Plan-And-Execute做宏观规划，ReAct做每一步的具体执行，效果最佳。

十、核心洞察

1. 200行代码揭示了Claude Code的核心原理

那些看起来神奇的AI编程助手，底层原理就是System Prompt + 工具 + ReAct循环。理解了这200行代码，你就理解了Agent的本质。

2. System Prompt是Agent的灵魂

不需要特殊训练模型，只需要精心设计Prompt，就能把普通LLM变成Agent。Prompt设计的质量直接决定Agent的表现。

3. 安全确认机制不可少

执行命令（execute_command）前必须让用户确认。Agent有能力删除文件、安装软件——没有安全机制的Agent是危险的。

4. 最大迭代次数防死循环

如果模型在某个步骤上反复失败，没有迭代上限的话Agent会无限循环。10次是个合理的默认值。

💡 关键认知：最重要的洞察是——两种模式不是非此即彼，而是可以组合。实际产品中，Plan-And-Execute负责宏观规划（"先做什么后做什么"），ReAct负责每一步的具体执行（"这一步该调什么工具"）。这就是Claude Code和Cursor等产品的真正架构。
