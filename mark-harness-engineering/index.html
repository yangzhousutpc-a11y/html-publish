📋 内容综述

这一期视频回答的是 2026 年 AI 圈最热的新名词——Harness Engineering——到底是什么。马克用"Agent = Model + Harness"这一行简洁公式开场，把 Harness 解释成"套在 LLM 这匹烈马身上的马具"：模型决定"想做什么"，Harness 决定模型能看到什么、能用什么工具、失败后怎么办。它不是一项新技术，而是把过去两年零散的 agent 工程实践第一次系统命名。

视频按"是什么 → 与谁有关 → 谁在干 → 从哪来 → 是不是炒作"五段式展开：先用层次图把 Prompt Engineering（2022-2023）→ Context Engineering（2024-2025）→ Harness Engineering（2026） 串成一条递进的研究范围扩展史；再分别拆解 Anthropic（Claude Code 的 nO 循环、Ralph Wiggum 双 Agent 架构、Managed Agents 三件解耦） 与 OpenAI（Codex 三层架构、AGENTS.md 极简化、subagent 工作流、百万行实验） 的实战路线，让读者直观感受到"同一个目标、两条工程路径"。

后半段，马克把镜头拉回起源争议：词是 2026 年 2 月被 Mitchell Hashimoto 命名的，但概念早在 2024 年 11 月就被 Anthropic 用过；OpenAI 在 2026-02-11 跟进发文一举引爆词条。最后回到价值判断：它是不是噱头？ 马克的结论与 Anthropic 一致——"不是噱头，但也不是终局"：随着模型变强，部分 harness 会被吞回模型权重；但 harness 也会变形以解锁更复杂任务，是 AI 工程过渡期的必修课。

一、Harness Engineering 是什么

1.1 一句话定义

把 LLM 比作一匹会推理的烈马，Harness 就是套在它身上的马具——工具、上下文管理、权限、验证、记忆、回滚机制。Harness Engineering 就是设计这套马具的工程实践。

1.2 核心公式（LangChain Harrison Chase 提出）

Agent = Model + Harness

它的含义是：Model 决定"会推理多深"；Harness 决定"推理能落地多远"——两者乘积才是 Agent 的真实能力。

📐 图 1：Agent = Model + Harness 的乘积关系

1.3 更精确的工程定义（Mitchell Hashimoto）

"每当 Agent 犯一次错，就花时间做工程改造，让它再也不犯同样的错。"

这把 Harness Engineering 从"形容词"变成了"动作"——它是一种反思 → 工程化 → 沉淀的迭代循环。

1.4 八大核心组件

#组件作用典型实现
1Prompt / 上下文组装把用户意图 + 系统指令 + 历史拼成一次推理输入XML 标签、模板引擎
2工具系统提供 read / write / bash / grep 等原语Anthropic 18 个原子工具
3执行循环while tool_call: 执行 → 观察 → 再调模型nO Loop、Ralph Loop
4Sandbox / 权限限定模型能动什么文件、能跑什么命令seatbelt、Landlock、容器

二、与 Prompt / Context Engineering 的关系

2.1 一句话

三者是研究范围层层向外扩展的递进关系，不是互斥而是包含。

2.2 三层递进对比

层关注点代表性问题时代
Prompt Engineering一句话怎么问"如何让模型 zero-shot 也答对？"2022-2023
Context Engineering整个上下文窗里塞什么、怎么压"RAG 后召回哪些片段？多轮如何压缩？"2024-2025
Harness Engineering模型之外的整套执行系统"Agent 失败后怎么自动恢复？"2026
📐 图 3：从 Prompt 到 Harness 的研究范围扩展

2.3 Harness Engineering 的关键洞察

"Harness engineering 是 context engineering 的延伸，趋势是把更多上下文控制权交给模型自己。" —— LangChain Harrison Chase

这背后是一个经济学逻辑：

  • 前沿模型差距越来越小（GPT / Claude / Gemini 顶尖模型只差几个百分点）；
  • 用得好不好不再取决于"换更强的模型"，而是取决于 harness 设计；
  • 因此 Harness Engineering 是工程红利时代的产物。

💡 关键认知：当所有人都在用同一档模型时，护城河就在 harness 里。这也是 Cursor、Devin、Codex 这些"看起来都用 GPT 系"的产品最终拉开差距的根因。

三、Anthropic 实战（Claude Code & Managed Agents）

3.1 一句话

Anthropic 的 harness 哲学是"简单循环 + 原子工具 + 严格验证"，相信"原语优于集成"。

3.2 Claude Code 的 nO 循环

  • 内部代号 nO，单一扁平消息列表，不做花哨编排；
  • 工具集只有约 18 个原子工具（Bash / Glob / Grep / LS / Read / Write / Edit / MultiEdit / WebSearch / WebFetch / TodoWrite / Task / ExitPlanMode 等）；
  • 哲学："原语优于集成"——选 ripgrep 正则而非向量数据库，让模型自己学会组合原语。
📐 图 4：Claude Code 的 nO 循环

3.3 Ralph Wiggum Loop（双 Agent 接力）

2024-11 Anthropic 首发，是"长链路 agent"的最早实践之一。

Agent职责持久化方式
Initializer Agent建脚手架、写 init 脚本、初始化 claude-progress.txt、做第一个 git commit把功能清单写进进度文件
Coding Agent每个 session 读 git log + 进度文件，做一个功能就提交用文件系统在 context window 之间传递记忆

💡 关键认知：context window 不能保存的东西，git 和文件系统能保存。Ralph Loop 用最朴素的 Unix 哲学解决了"agent 失忆"问题。

3.4 三 Agent GAN 架构（2026-03）

Agent职责关键设计
Planner拆需求成子任务清单不写代码，只规划
Generator根据计划写代码拥有完整工具集
Evaluator独立评估，避免"王婆卖瓜"完全隔离上下文，看不到 Generator 的"自圆其说"
📐 图 5：三 Agent GAN 架构 vs Solo 方案的实证对比

3.5 Context Reset vs Compaction（解决 Context Anxiety）

  • Context Anxiety（上下文焦虑）：Sonnet 4.5 出现的现象——模型感觉上下文快满就草草收尾；
  • Compaction：摘要续写；
  • Context Reset：清空重启，靠 session.log 与文件系统恢复状态。

💡 关键认知：Opus 4.5 之后这个问题消失了——也就是说 Context Reset 这个 harness 组件被模型升级"吃掉"了。这是 Harness Engineering 最魔幻也最焦虑的一面：你今天精心设计的脚手架，明天可能就被模型权重吸收。

3.6 Managed Agents（2026）三件解耦

[ Brain ]    = Claude + harness
[ Hands ]    = sandbox / 容器 / 工具
[ Session ]  = append-only 事件日志

任何组件挂掉都能用 wake(sessionId) 重启——从"养宠物"变成"养牛"（pets vs cattle）。

5.2 关键时间线

5.3 词根来源

  • Test Harness：传统软件工程的"测试外壳"；
  • lm-evaluation-harness：开源 LLM 评测框架；
  • 两者合流，借给了 agent 工程。

💡 关键认知：Harness Engineering 不是凭空发明的新词——它是老术语在新语境下的复用。这恰恰说明 agent 工程在向传统软件工程靠拢，而不是离开。

六、是噱头还是技术突破

6.1 一句话

不是噱头，但也不是终局——它是 AI 工程过渡期的必修课。

6.2 质疑方观点

论点出处
没有新技术——linter、任务拆解、评估全是旧工具旧瓶装新酒HN 顶贴
模型变强会自动吸收 harness（Sonnet 4.5 → Opus 4.6 减少了 harness 复杂度）Anthropic 自己承认
"是不是只是 retry logic 换个名字？"HN 评论

6.3 支持方观点

论点数据 / 出处
实证有效：同模型仅换 harness，效果显著提升Archon PR 接受率 6.7% → 70%；LangChain Deep Agents 在 Terminal-Bench 2.0 把 52.8% → 66.5%
第一次系统化命名了零散实践，让经验可传承LangChain、Anthropic 多篇 blog
Cursor 训练 Composer 时把 harness 编进模型权重——证明它是真实抽象层Cursor 工程博客
📐 图 8：Harness Engineering 价值判断的两面

6.4 马克的判断（与 Anthropic 一致）

"不是噱头，但不是终局；模型进化会吞掉一部分 harness，但 harness 会变形以解锁更复杂任务。"

Anthropic Managed Agents 那句话最妙：

"Harness encode assumptions that go stale as models improve."
（Harness 编码的是会随模型升级而过期的假设。）

它本就该是会过期的脚手架。

💡 关键认知：判断一个 AI 概念是不是炒作，可以问三个问题：
① 它能否改善实证指标（不只是新词）？
② 它能否沉淀团队经验（不只是个人技巧）？
③ 它能否经受模型升级冲击（不只是某代模型的临时补丁）？
Harness Engineering 在前两点上是 yes，第三点上是 "yes，但形态会变"——因此它是过渡期的必修课，不是终局答案。

七、结语：从工具到方法论

马克在视频末尾给出了一个升维的总结：

  • Prompt Engineering 教会我们"怎么和模型说话"；
  • Context Engineering 教会我们"给模型什么饭吃"；
  • Harness Engineering 教会我们"让模型生活在什么样的环境里"。

这三层是同一个方法论的三层皮：随着模型能力上升，工程师关注的边界不断外移——从一句话、到一个窗口、再到整个执行系统。下一个会是什么？也许是 Society Engineering——多个 agent 之间的社会规则与协作协议。
