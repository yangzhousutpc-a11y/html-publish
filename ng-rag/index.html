📖 内容综述

RAG（检索增强生成）解决了LLM最大的痛点：知识会过时、会幻觉、不懂你公司的内部文档。这门课教的不是"怎么做一个Demo版RAG"，而是"怎么从Demo升级到生产级"——核心方法是高级检索技术 + 自动化评估框架。

课程按升级路径递进：先讲基础RAG流程（文档→分块→嵌入→检索→生成）让你知道全貌；然后深入分块策略（智能分块解决40%的检索故障）；再引入混合检索+重排序的"王炸组合"；接着讲查询扩展（HyDE/多查询/步回提示）优化用户查询；最后用三维评估框架（忠实度/相关性/正确性）量化系统表现。

一句话概括：分好块、搜得全、排得准、评得住。

一、为什么需要RAG？

一句话：LLM的知识有截止日期、会幻觉、不懂你的私有数据——RAG让它在回答前先"查资料"，有据可依。

LLM的问题 RAG的解决方案
知识截止（不知道昨天发生的事） 实时检索最新文档
幻觉（编造事实） 基于检索到的文档回答
不懂私有数据（公司内部文档） 接入企业知识库

二、基础RAG流程

【摄入阶段】
文档 → 分块(Chunking) → 嵌入向量化(Embedding) → 存入向量数据库

【检索阶段】
用户查询 → 向量化 → 在向量数据库中搜索相似内容 → 取Top-K相关块

【生成阶段】
检索结果 + 查询 → 组成Prompt → LLM生成回答

💡 关键认知：RAG的核心洞察是——检索质量决定了生成质量。如果检索到的文档段落不相关，哪怕LLM再强也回答不好。这就是为什么RAG的80%精力应该花在"怎么检索得更准"上，而不是"换一个更强的LLM"。

三、分块策略：解决40%检索故障

分块参数的影响

参数 太小 推荐值 太大
Chunk大小 100-200token（缺上下文） 300-500token 1000+token（引入噪声）
重叠(Overlap) 0（答案卡在边界） ~100token —

课上的关键数据：智能分块可以解决约40%的检索故障。

三种分块策略

策略 原理 推荐度
固定大小 按字符数切 简单但粗暴
语义分块 按含义切 效果好但成本高
递归分割 ✅ 先按段落→再按句子→最后按字符 主流选择

💡 关键认知：别小看100token的重叠——它防止了答案刚好被切断在两个chunk边界的情况。很多"明明有文档但检索不到"的问题，加个重叠就解决了。

四、高级检索策略

基础RAG的检索太粗暴——直接向量搜索Top-K就完事。高级策略从四个方向优化：句子窗口（精准+上下文）、自动合并（碎片→完整段落）、混合检索（向量+关键词双保险）、重排序（Cross-Encoder精排）。

句子窗口检索

问题：基础RAG检索到的单个句子上下文不足。 方案：检索最相关的句子，但返回它前后的完整段落作为"窗口"。

📐 句子窗口检索: 精准定位+扩大上下文

自动合并检索

问题：文档切得太碎，信息碎片化。 方案：构建树状文档结构（子节点=小chunk，父节点=合并文本）。当多个相邻子节点命中时，自动返回它们的父节点。

文档树：
       [父节点：完整段落]
      /    |    |    \
   [子1] [子2] [子3] [子4]

如果子1、子2、子3都命中 → 自动返回父节点
（实现"细节"和"全貌"的自动平衡）

混合检索：向量+关键词的"王炸"

为什么单一检索会失效？

方式 擅长 短板 失效例子
纯向量 语义理解 精确匹配差 "X-2000处理器"、"Error 503"
纯BM25 关键词匹配 语义理解差 "不含糖的饮料"（否定逻辑）

混合方案：同时跑两路搜索，用RRF算法融合结果。

RRF（倒数排序融合）： - 不依赖原始分数（因为两路分数量纲不同） - 仅基于排序位置：score = Σ 1/(60+rank) - 某文档在两个列表都排第1，得分最高

📐 高级检索策略: 向量+关键词+重排序

重排序：Cross-Encoder精排

问题：混合检索提升了召回率，但Top-K中仍混杂无关内容。

方案：用Cross-Encoder对候选文档逐一精排。

阶段 策略 数量 特点
L1 粗筛 混合检索 Top 20-50 快速、便宜、召回全
L2 精排 Cross-Encoder Top 3-5 慢但精准

课上数据：引入重排序后，正确chunk出现在前3的命中率从68%提升至89%。

💡 关键认知："混合检索+重排序"是生产级RAG的王炸组合——混合检索解决"找得全"，重排序解决"排得准"。两者缺一不可。

基础RAG实现（LangChain示例）

from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma

# 1. 加载文档
loader = PyPDFLoader("公司手册.pdf")
docs = loader.load()

# 2. 分块（递归字符分割，推荐）
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,    # 每块512字符
    chunk_overlap=50   # 相邻块重叠50字符
)
chunks = splitter.split_documents(docs)

# 3. 向量化 + 存储
db = Chroma.from_documents(chunks, OpenAIEmbeddings())

# 4. 检索 + 生成
results = db.similarity_search("公司的年假政策是什么？", k=3)
# 将results注入LLM的Context → 生成精准回答

五、查询扩展与转换

多查询检索

用LLM生成查询的3个不同表述变体，分别检索后合并结果，提高召回率。

HyDE（假设文档嵌入）

反直觉但有效：先让LLM生成一段"假设的答案文档"（即使可能有错），用它的向量去检索真实文档——因为短查询和长文档的语义空间不匹配，假设文档作为"桥梁"。

步回提示（Step-Back）

让LLM先"退一步"思考更抽象的问题，扩大检索范围获取更完整的背景信息。

六、RAG评估三维框架

三个维度

维度 评估什么 防止什么
上下文相关性 检索到的文档和问题相关吗？ 检索到无关文档
忠实度/依据性 回答是基于检索文档的吗？ 模型编造（幻觉）
答案相关性 回答解决了用户的问题吗？ 答非所问

评估工具：TruLens

💡 关键认知：很多RAG系统只看"答案好不好"，忽视了"检索准不准"和"有没有幻觉"。三维评估才能全面把握——检索质量差但答案看起来对，可能只是模型靠"记忆"在回答，一换问题就露馅。

七、生产级RAG升级路径

Level 1: 基础RAG（固定分块+向量检索）
    ↓ 解决~40%问题
Level 2: 智能分块+元数据管理
    ↓ 解决召回问题
Level 3: 混合搜索（向量+BM25）
    ↓ 解决精度问题
Level 4: 重排序（Cross-Encoder）
    ↓ 解决复杂查询
Level 5: 高级检索（句窗/自动合并/查询扩展）
    ↓ 解决安全问题
Level 6: 护栏（阈值检查+冲突检测+约束指令）

护栏机制： - 最高置信度 < 0.6 → 拒绝回答，要求用户重新表述 - 来源日期跨度大 → 自动添加"最新政策优先"警告 - Prompt中明确要求"仅基于上下文回答，不足时说明"
