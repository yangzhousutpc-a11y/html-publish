📋 内容综述

这个视频讲了一个AI领域正在发生的重要概念演进：从提示词工程到上下文工程再到驾驭工程（Harness Engineering），三者不是替代关系，而是逐层嵌套升级。Prompt Engineering 管"说什么"，Context Engineering 管"知道什么"，Harness Engineering 管"在什么环境里做事"。这背后映射的是人类对AI认知的升级——AI从无状态工具，到需要上下文的助手，再到需要被驾驭的自主Agent。

一、提示词工程：跟AI"说什么"

1.1 提示词工程的诞生

2020年GPT-3发布，Brown等人的论文证明：同一模型只需改变prompt写法，就能变成翻译器、摘要器、代码生成器。提示词工程从此成为独立学科。2022年底ChatGPT发布后，"Prompt Engineer"成了硅谷最火职位，年薪30万美金起。

1.2 核心技巧体系

技巧核心思想适用场景
Zero-shot仅用指令，不给示例简单分类、生成
Few-shot给1-几个示例引导输出格式风格对齐、格式约束
Chain-of-Thought诱导模型展示中间推理步骤多步推理、数学
Self-Consistency多次采样取共识减少随机性
Tree-of-Thought分支搜索多条推理路径复杂规划
ReAct推理+行动交替需要工具调用的任务
💡 关键认知：2024年后，随着Claude 3.5 Sonnet等模型出现，Prompt技巧的边际收益急速下降。模型足够聪明时，怎么问已经没那么重要，重要的是问的时候它手里有没有关于你问题的足够且合适的信息。

1.3 提示词工程的天花板

  • 上下文腐烂（Context Rot）：上下文窗口填充率超过40%时，输出质量快速衰退
  • 幻觉问题：模型没有的信息它就是没有，无论怎么措辞都没用
  • 单轮交互局限：Prompt工程优化的是单次交互质量，Agent多步推理时力不从心

二、上下文工程：给AI"喂什么信息"

2.1 谁提出的？

时间人物核心观点
2025年6月Shopify CEO Tobi Lütke"I really like the term 'context engineering' over prompt engineering."
2025年6月前OpenAI联创 Andrej Karpathy"Context engineering is the delicate art and science of filling the context window with just the right information for the next step."
2025年7月Gartner"Context engineering is in, and prompt engineering is out"
2025年10月Anthropic发布《Effective Context Engineering for AI Agents》系统阐述

2.2 核心定义

上下文工程是设计和构建动态系统的学科，能够在恰当的时机、以恰当的格式，提供恰当的信息和工具，让LLM拥有完成任务所需的一切。

与提示词工程的核心区别：

  • 提示词工程：How should I phrase this?（怎么写指令）
  • 上下文工程：What does the model need to know right now?（模型现在需要知道什么）

2.3 五大核心策略

策略含义典型技术
Offload（转移）将不必要的信息移出上下文外部存储、工具代理
Reduce（压缩）压缩上下文信息摘要、Compact
Retrieve（检索）按需检索相关信息RAG
Isolate（隔离）分离不同任务的上下文子Agent、独立会话
Cache（缓存）缓存高频上下文减少重复计算Prompt Caching

2.4 六大技术组件

  1. 检索增强生成（RAG）：运行时从外部知识库检索相关文档注入上下文
  2. 记忆系统：工作记忆→会话状态→长期记忆的多层级体系
  3. 动态上下文决策：根据当前步骤，主动选择并注入精准信息
  4. 压缩与摘要：长对话超出窗口时触发Compact
  5. Token预算管理：在有限窗口中分配竞争性需求
  6. 工具编排：选择模型可访问的工具、定义输入输出格式
💡 关键认知：上下文可以告诉Agent"知道什么"，但无法阻止Agent"做不该做的事"。这是上下文工程的关键悖论——也是Harness Engineering诞生的根本原因。

三、Harness Engineering：让AI在什么环境里做事

3.1 驾驭工程的核心隐喻

Harness 一词本意为"马具/挽具"——马鞍、缰绳、嚼子那一整套用来驾驭马匹的装备。

马是一种非常强大的动物，速度快、力量大，但如果不给它套上缰绳，它大概率会跑偏甚至把你甩下来。Harness 的作用是把这股野蛮的力量引导到需要的方向上。AI Agent 就是那匹马，Harness 就是缰绳。

📐 Harness 核心隐喻：AI Agent = 马，Harness = 缰绳

3.2 概念提出时间线

时间事件
2025年11月Anthropic 发布 Claude Agent SDK，称为"a powerful general-purpose agent harness"
2026年2月5日HashiCorp创始人 Mitchell Hashimoto 在博客首次提出"Engineer the Harness"概念
2026年2月11日OpenAI 官方博客《Harness Engineering》，彻底将这个词带入主流
Mitchell Hashimoto 的经典定义："每当你发现Agent犯了一个错误，你就花时间设计一个解决方案，使Agent永远不再犯同样的错误。"
OpenAI 的核心论断："Humans steer. Agents execute."（人类掌舵，智能体执行）

3.3 正式定义与核心公式

Harness Engineering 是指在AI Agent运行环境中，设计和构建约束机制、反馈回路、工作流控制以及持续改进循环的系统工程实践。它不优化模型本身，而是优化模型运行的环境。

Agent = Model + Harness

Model 提供推理能力（CPU）
Harness 提供方向盘和刹车（操作系统）

3.4 两类控制机制

📐 两类控制机制：Guides + Sensors
💡 关键认知：好的Harness = Guides + Sensors，前者防患未然，后者亡羊补牢，形成闭环。为了获得更高的AI自主性，运行时必须受到更严格的约束。增加信任需要的不是更多自由，而是更多限制。

3.5 四种Agent失败模式

失败模式表现根因
一步到位综合症试图在单个上下文窗口完成全部工作，消耗大半后开始幻觉上下文管理不当
过早宣布胜利编译都通不过就说"编码完成"缺乏验证机制
过早标记功能完成功能已实现但未端到端验证缺乏全链路测试
冷启动问题多次会话间缺乏持久化记忆缺乏状态持久化

根本原因：Agent无法准确评估自身产出质量。

4.4 从0到1的落地顺序

  1. 写 CLAUDE.md / AGENTS.md → 定义Agent能看到什么
  2. 编码化约束 → 规则进Linter / CI / 类型系统，不靠Agent自觉
  3. 接工具 → MCP接外部系统，Skills定义使用流程
  4. 加Middleware → Hook在关键节点插入确定性逻辑
  5. 建评测 → 每个真实失败转成测试用例
  6. 闭环 → Agent失败 → 更新文档 → 下次不再犯

最小可用Harness（如果只能做三件事）：

  1. 写一份好的 CLAUDE.md
  2. 跑自动化测试
  3. 建立反馈循环
