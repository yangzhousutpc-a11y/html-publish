📖 内容综述

这份笔记用一条完整的技术链路串起了AI领域最核心的8个概念：LLM → Token → Context → Prompt → Tool → MCP → Agent → Agent Skill。每个概念都不是孤立的知识点，而是上一个概念的自然延伸——LLM是大脑，Token是大脑处理信息的最小单位，Context是大脑的短期记忆，Prompt是和大脑说话的方式，Tool是给大脑装上手脚，MCP是让手脚通用的USB-C接口，Agent是大脑+手脚+自主循环的完整体，Skill则是把Agent的最佳实践沉淀成可复用的经验手册。

从头看到尾，你会发现这8个概念就像搭积木一样，一层叠一层，最终搭出了一个能自主干活的AI助手。

一、LLM：超级文字接龙机

一句话：大语言模型就是一个"文字接龙"高手——你给它开头几个字，它根据概率预测下一个最可能的字，一个接一个地"接龙"出完整的回答。

它到底在做什么？

想象你在玩文字接龙游戏。你说"张三的视频"，然后模型开始预测：

输入："张三的视频"
模型预测：怎么样 → 很 → 有 → 趣 → 。

每一步，模型都不是在"理解"你说的话，而是在计算"根据前面这些字，下一个字最可能是什么"。它看过了互联网上几千亿个词的语料，学会了语言的统计规律。

📐 LLM "文字接龙" 预测流程

LLM和普通程序有什么区别？

维度 传统程序 LLM
工作方式 按规则执行（if-else） 按概率预测
输出特点 确定的 每次可能不同
知识来源 程序员手写规则 训练数据中学习
能处理的任务 特定任务 几乎所有语言任务

💡 关键认知：LLM不是数据库，不是搜索引擎——它是一个概率函数。它的每一个输出都是"掷骰子"的结果，只不过这个骰子被训练得非常精准。这就是为什么它有时候会"一本正经地胡说八道"（幻觉）。

二、Token：模型眼中的"原子"

一句话：Token是大模型处理文本的最小单位——不是字，不是词，而是分词器切出来的"碎片"。

从LLM到Token的递进

上一节说LLM是"一个字一个字地预测"，但准确地说，它是一个Token一个Token地预测。Token是模型能看到的最小颗粒。

Token ≠ 词，这是最常见的误区

英文："information" → 可能被切成 ["infor", "mation"] = 2个Token
中文："我喜欢苹果" → 可能被切成 ["我", "喜欢", "苹果"] = 3个Token

📐 Token 分词对比 — 同一句话在模型眼中的"切法"

中英文Token换算（实用）

语言 换算关系 意味着
英文 1 Token ≈ 0.75个单词 100个词 ≈ 133 Token
中文 1 Token ≈ 1.5-2个汉字 100个字 ≈ 50-67 Token

💡 关键认知：模型的计费、Context长度限制、响应速度——全部以Token为单位。理解Token，后面所有概念才有根基。这也是为什么用中文提问比英文"更烧钱"的原因。

三、Context：模型的"短期记忆"

一句话：Context就是每次发给模型的信息总和——包括你之前说的、系统预设的、工具返回的，统统打包成一大段文字扔给模型。

从Token到Context的递进

Token是信息的最小单位，那么一次对话中包含的所有Token加在一起，就构成了Context。模型就是在这个Context的范围内进行预测。

为什么模型能"记住"之前的对话？

答案：它根本不记得。

每次你发消息，系统都会把之前的完整对话历史重新发一遍给模型。模型"记住"对话的方式，就像你每次考试都要重新看一遍所有笔记。

📐 Context 的秘密 — 每次对话都是"重新打包"

第1轮：用户说"你好" → 发给模型 ["你好"]
第2轮：用户说"我叫张三" → 发给模型 ["你好", "你好呀！", "我叫张三"]
第3轮：用户说"我叫什么？" → 发给模型 ["你好", "你好呀！", "我叫张三", "你叫张三", "我叫什么？"]

Context Window：记忆容量的上限

模型 Context Window 大约能"记住"多少
GPT-3.5 4K/16K Token 几页纸
Claude 3 200K Token 一本小说
GPT-4o 128K Token 半本小说

超过上限怎么办？最早的对话记录会被"丢弃"——这就是长对话后模型"忘记"你说过的话的原因。

💡 关键认知：Context有上限，这直接引出了下一个问题——如果我想让模型知道公司内部的100页文档怎么办？答案就是RAG（检索增强生成）：不把所有文档塞进去，而是每次只把最相关的几段塞进Context。

四、Prompt：和模型"说话"的艺术

一句话：Prompt就是你发给模型的指令。但指令分两种——System Prompt是开发者设定的"人设"，User Prompt是用户的具体问题。

从Context到Prompt的递进

Context是信息的容器，而Prompt是放进容器里最关键的那部分——它决定了模型会给你什么样的回答。

System Prompt vs User Prompt

视频中的"数学辅导机器人"例子：

不加System Prompt：

用户：3x + 5 = 20，求x
模型：x = 5（直接给答案，冰冷的数学回答）

加上System Prompt后：

System Prompt："你是一个耐心的数学辅导老师，要用通俗易懂的方式讲解，引导学生思考，而不是直接给答案。"

用户：3x + 5 = 20，求x
模型：同学你看呀，我们先把等号两边的5挪走...（一步步引导）

同一个模型，同一个问题，仅仅是换了一段System Prompt，输出就天壤之别。

Prompt工程的四层架构

📐 Prompt 四层架构图

层级 作用 例子
角色定义 设定身份 "你是资深工程师"
任务描述 明确目标 "开发一个API"
约束条件 限制格式/风格 "使用OpenAPI规范"
示例参考 提供输入输出样本 给出1-2个示例

💡 关键认知：好的Prompt就是"说话方式"的改变——无需改动模型参数，仅通过精心设计指令，就能让同一个模型表现出天壤之别。这就是为什么Prompt Engineering成为AI时代的核心技能。

五、Tool：给模型装上"手脚"

一句话：Tool就是模型能调用的外部函数——查天气、读文件、执行代码——让模型从"只会说"变成"能做事"。

从Prompt到Tool的递进

Prompt让模型知道"要做什么"，但模型本身只会生成文字。查天气？它不知道真实天气。写文件？它没有文件系统。Tool解决的就是这个问题。

天气查询的完整调用流程

这是视频中最经典的例子：

① 用户："明天北京会下雨吗？"
② 模型思考：我需要查天气，调用 get_weather 工具
③ 模型请求：get_weather(city="北京", date="明天")
④ 系统执行：调用真实天气API，获得"明天北京有雨，80%"
⑤ 结果返回给模型
⑥ 模型回答："根据天气预报，明天北京有80%概率下雨，建议带伞。"

📐 Tool调用完整六步流程

工具描述的JSON格式

模型怎么知道有哪些工具可用？靠的是结构化的工具描述：

{
  "name": "get_weather",
  "description": "获取指定城市的天气预报",
  "inputSchema": {
    "type": "object",
    "properties": {
      "city": { "type": "string" },
      "date": { "type": "string" }
    },
    "required": ["city", "date"]
  }
}

💡 关键认知：模型是"指挥官"，不是"执行者"。它负责决策（"我要查天气"）和参数选择（"城市=北京"），但真正执行查询的是外部系统。Tool让模型从"知道"变成"能做"——没有Tool的LLM就像被困在书房里的人。

六、MCP：AI世界的USB-C接口

一句话：MCP（Model Context Protocol）是工具的"万能转接头"——开发者只需写一次工具代码，就能在所有支持MCP的AI平台上使用。

从Tool到MCP的递进

Tool解决了"模型能用工具"的问题，但新问题来了——每个AI平台（ChatGPT、Claude、Gemini）调用工具的方式都不一样！MCP就是为了解决这个碎片化问题。

USB-C比喻（视频核心比喻）

MCP之前 = 充电器碎片化时代
├─ iPhone用Lightning
├─ 某安卓用Micro USB
└─ 另一个安卓用Type-C
→ 每换一台手机就要买新线

MCP之后 = USB-C统一时代
└─ 所有设备统一用USB-C
→ 一条线到处用

对应到AI领域： - MCP之前：给ChatGPT写一套工具集成代码，给Claude再写一套，给Gemini又写一套 - MCP之后：写一次MCP Server，所有平台都能用

MCP的三层结构

角色 是什么 例子
MCP Host 支持MCP的AI应用 Claude Desktop、Cursor、Cline
MCP Server 提供工具的程序 天气服务、数据库服务
Tool Server内部的具体功能 get_weather、get_alerts

📐 MCP 三层架构图

💡 关键认知：MCP的价值不在Tool本身，而在于统一标准。它让整个AI工具生态从"各自为战"变成"一次开发，到处可用"。MCP是连接Tool和AI平台的桥梁，而不是替代Tool。

七、Agent：自主干活的智能体

一句话：Agent = LLM + 工具 + 循环执行。它不是一问一答的聊天，而是"接到任务后自己拆解步骤、调用工具、直到完成目标"的自主执行者。

从MCP到Agent的递进

有了MCP，模型可以方便地使用各种工具。但问题是——谁来决定"什么时候用什么工具"？答案是Agent。Agent就是那个自主做决策的"人"。

和普通对话的区别

维度 普通对话 Agent
输入 用户问题 用户目标/任务
输出 一段回答 完整的执行结果
过程 一次性回复 多步骤循环执行
自主性 被动回应 主动规划和执行

"下雨查伞店"的完整执行链

用户："明天下雨，帮我找个地方买伞"

Agent第1轮：
  思考：我需要确认明天是否真的下雨
  行动：调用 get_weather("当前城市")
  观察：确认明天下雨

Agent第2轮：
  思考：下雨确认了，现在找附近的伞店
  行动：调用 search_shops("雨伞", "当前位置")
  观察：找到3家伞店

Agent第3轮：
  思考：有伞店了，看看哪家在营业
  行动：调用 check_hours(shop_id)
  观察：XX店现在营业

最终输出："明天确实会下雨。推荐XX伞店，现在营业，地址在..."

📐 Agent "下雨查伞店" 完整执行链

关键发现：整个过程中，用户只说了一句话。是Agent自己决定要查天气、找伞店、确认营业——这就是"自主性"。

💡 关键认知：Agent的强大在于它的"自主循环"——不是我告诉它"先查天气再查伞店"，而是它自己意识到需要这样做。这就是从"工具"到"助手"的跨越。Agent = LLM(大脑) + Tool(手脚) + Loop(持续反馈)。

八、Agent Skill：可复用的经验手册

一句话：Skill是Agent的"标准操作手册"——把特定领域的最佳实践（指令+脚本+参考资料）打包成一个可复用的文件夹，让Agent在任何场景下都能像专家一样工作。

从Agent到Skill的递进

Agent能自主干活了，但每次都要从零开始写Prompt、定义工具。Skill就是把"经过验证的最佳实践"沉淀下来——下次遇到同类任务，直接加载Skill就行。

三层渐进式披露（Token经济学设计）

这是Skill最精妙的设计。为什么不把所有信息一次性扔给Agent？因为Context Window有限、Token要钱。所以Skill分三层按需加载：

层级 内容 加载时机 Token消耗 类比
元数据 Skill的名字和简介 始终加载 极少 员工简历
指令 完整的SKILL.md正文 选中后加载 中等 操作手册
资源 Reference文档+Script脚本 按需加载 可选 参考附录

📐 Skill 三层渐进式披露机制

SKILL.md文件结构

my-skill/
├── skill.md          ← 必须：元数据 + 指令正文
├── references/       ← 可选：按条件加载的长文档
│   └── 财务手册.md
├── scripts/          ← 可选：可执行脚本
│   └── upload.py
└── examples/         ← 可选：输入输出样例

"出门清单助手"完整例子

第一层（始终加载）： - name: travel-checklist - description: "根据行程生成出门清单"

第二层（选中后加载）： - 基础规则：根据目的地和季节生成清单 - 例如：北京冬天 → "羽绒服、暖宝宝、保温杯"

第三层（按需加载）： - Reference：提到"商务会议" → 加载《商务着装规范.md》 - Script：要求"发送到邮箱" → 执行 send_email.py

Reference vs Script的关键区别

维度 Reference Script
本质 读文件进Context 跑代码，代码不进Context
Token消耗 有（与文件大小成正比） 几乎为零（仅结果的Token）
用途 规章手册、知识库 上传文件、API调用、自动化

💡 关键认知：Skill的出现，让AI从"一个聊天机器人"变成了"一套标准化的工具集"。它解决了企业级痛点——没有Skill的Agent像每天失忆的员工，有了Skill就能标准化执行、稳定输出。

核心概念递进图

📐 8 个核心概念完整递进链路

每一层都解决了上一层遗留的问题： - Token让LLM能处理信息 → 但处理范围有限 - Context扩大了信息范围 → 但如何精准指挥？ - Prompt解决了指挥问题 → 但模型只能说不能做 - Tool让模型能做事 → 但每个平台不通用 - MCP统一了标准 → 但谁来自主决策？ - Agent实现了自主性 → 但每次从零开始 - Skill沉淀了最佳实践 → 形成可复用的能力
