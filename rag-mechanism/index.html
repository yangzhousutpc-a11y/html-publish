RAG 工作机制详解 学习笔记

课程：马克的技术工作坊 B站视频：BV1JLN2z4EZQ（单集，约17分钟） 讲师：马克 适合人群：想理解RAG底层原理的AI开发者和产品经理

📖 内容综述

这份笔记用一条完整的技术链路，把RAG（检索增强生成）从"为什么需要"到"怎么做到"讲透了。17分钟，5个核心环节，一个端到端的闭环。

先回答"为什么"——LLM有三大硬伤（知识截止、不懂私有数据、上下文塞不下），RAG用"先检索再生成"一招解决。然后进入核心流程：分片把长文档拆成碎片，索引把碎片变成向量存进数据库，召回用向量相似度粗筛出Top 10，重排用Cross-Encoder精筛到Top 3，最后生成把3个精选片段+问题扔给LLM得到精准回答。

最精妙的比喻是"招聘流程"——召回=简历初筛（快但粗），重排=高管面试（慢但准）。两层筛选组合起来，既快又准。

一、为什么需要RAG？

一句话：LLM再强也有三大硬伤——RAG让它在回答前先"查资料"，有据可依。

LLM的三大硬伤

硬伤 症状 RAG的解法
知识截止 不知道训练数据之后发生的事 实时检索最新文档
私有数据盲区 完全不懂你公司的产品手册 接入企业知识库
上下文塞不下 上千页文档直接扔进去会爆 只检索最相关的几段

📐 LLM三大硬伤 → RAG的解法

视频中的具体场景

假设你要做一个智能客服，回答公司产品问题：

不用RAG的做法：
把上千页产品手册直接扔给LLM
→ 超出上下文限制 ❌
→ 推理成本爆炸 ❌
→ 速度极慢 ❌
→ "读了后面忘了前面" ❌

用RAG的做法：
把手册切成片段→向量化→存进数据库
用户问"产品X的价格？" → 只检索相关的3-5个片段
→ 快速准确回答 ✅
→ 成本可控 ✅

💡 关键认知：RAG解决的不仅是"知识问题"，更是"经济问题"——把上千页文档全塞进Context，每次调用的Token费用可能是检索方式的100倍。RAG让你只为"有用的信息"付费。

二、RAG的两阶段架构

一句话：RAG分"离线建库"和"在线问答"两个阶段，5个环节组成完整闭环。

【提问前·离线建库】
原始文档 → ①分片 → ②索引（向量化+存储） → 知识库就绪

【提问后·在线问答】
用户提问 → ③召回（粗筛Top 10） → ④重排（精筛Top 3） → ⑤生成（LLM回答）

接下来逐个拆解5个环节。

📐 RAG两阶段·五环节完整架构

三、环节①：分片（Chunking）

一句话：把长文档切成小片段——因为后续所有处理都基于"片段"而非"整篇文档"。

为什么必须分片？

  • 模型一次性处理几千页文本会崩溃
  • 检索需要精确到"段落级别"，而不是返回整本书
  • 分片后才能为每个片段生成独立的向量

分片策略

策略 做法 适用场景
按字数切 每1000字一个片段 简单粗暴，通用
按段落/章节切 按文档自然段落切分 结构化文档
按页码切 每页一个片段 PDF文档
递归分割 先按段落→再按句子→最后按字符 推荐方式

💡 关键认知：分片质量直接决定后续检索质量。切太大→检索时混入噪声；切太小→丢失上下文。推荐Chunk Size 300-500 Token，重叠(Overlap) ~100 Token防止答案被切断在边界。

四、环节②：索引（Indexing）

一句话：把文本片段变成数字向量，存进专门的数据库——这是RAG能"理解语义"的关键。

三个核心概念（理解RAG的灵魂）

概念1：向量（Vector）

向量就是一组数字，代表"方向+大小"：

二维向量：[1.0, 2.0]（平面上的一个点）
高维向量：[0.12, -0.45, 0.89, ...] （几百到几千维，包含丰富的语义信息）

概念2：Embedding（嵌入）

把文本变成向量的过程——而且神奇的是，语义相近的文本，向量在空间中距离也近。

视频中的具体例子：

"马克喜欢吃水果" → 向量 [1, 2]
"马克爱吃水果"   → 向量 [1, 1]  ← 距离很近 ✅ 语义接近
"天气真好"       → 向量 [-3, -1] ← 距离很远 ❌ 语义无关

💡 关键认知：Embedding的神奇之处在于——不需要手工定义"什么叫相似"，模型自动学习：语义接近的文本→向量接近。这是一种可迁移的通用能力。选模型可以参考MTEB排行榜。

📐 Embedding的魔法: 语义相近→向量距离近

概念3：向量数据库（Vector Database）

专门存储和查询向量的数据库。必须存两列：

原始文本（给LLM读的） 向量（给机器算相似度的）
"产品X价格为999元" [0.12, -0.45, 0.89, ...]
"产品X适用人群是..." [0.11, -0.43, 0.91, ...]

常见产品：Chroma（轻量）、Pinecone（云端）、Weaviate、Milvus

📐 向量数据库存两列: 原文(给LLM读) + 向量(给机器算)

索引的完整操作

片段1 → Embedding模型 → 向量1 ┐
片段2 → Embedding模型 → 向量2 ├→ 全部存入向量数据库
片段3 → Embedding模型 → 向量3 ┘
...
→ 知识库构建完毕，等待用户提问

五、环节③：召回（Retrieval）

一句话：用户提问后，在向量数据库中搜索最相关的片段——快速粗筛。

操作流程

用户提问："马克喜欢吃什么？"
     ↓
通过同一个Embedding模型转换为向量
     ↓
在向量数据库中计算 问题向量 与 所有片段向量 的相似度
     ↓
选出相似度最高的 Top 10 个片段

三种向量相似度计算方法

方法 原理 直觉理解 适用
余弦相似度 ⭐ 两向量夹角的余弦值 "两个方向有多接近？" 最常用，不受长度影响
欧式距离 两点间直线距离 "两个点在空间中有多远？" 对绝对位置敏感
点积 方向+长度的综合 "一个向量推另一个能推多远？" 综合考量

召回的特点： - ✅ 成本低、速度快（毫秒级） - ⚠️ 准确率相对较低（可能混入"看起来相似但无关"的片段）

💡 关键认知：召回阶段追求的是"宁可多选，不能漏选"（高召回率）。宁可选进来几个不太相关的，也不能把正确答案漏掉。精度问题交给下一步的重排来解决。

六、环节④：重排（Reranking）

一句话：从召回的Top 10中精选Top 3——用更精准但更慢的模型做二次筛选。

📐 三种向量相似度计算方法

为什么需要重排？

召回只靠向量相似度，会有"虚假相似"的问题。重排用Cross-Encoder做更深度的语义理解。

维度 召回（粗筛） 重排（精筛）
模型 Embedding（Bi-Encoder） Cross-Encoder
输入方式 问题和片段分别编码 问题+片段成对输入
速度 快（毫秒级） 慢（秒级）
准确率 相对较低 高
成本 低 高
输出 Top 10候选 Top 3精选

📐 招聘比喻: 召回(简历初筛) → 重排(高管面试)

招聘流程比喻（视频核心比喻）

召回 = 简历初筛（HR粗筛）
├─ 从成千上万份简历中筛出10人
├─ 只看学历、工作年限等关键词
├─ 快速、成本低
└─ 可能混入"简历好看但实际不行"的人

重排 = 高管面试（深度精筛）
├─ 对这10人逐一深度面试
├─ 全面评估沟通、专业、匹配度
├─ 耗时长、成本高
└─ 选出3个最合适的进入offer阶段

为什么不一开始就用面试？ 对上千人逐一面试，成本不现实。 为什么不只用简历筛？ 太容易选错人。 所以需要两层：先粗筛排除明显不合适的，再精筛找到最优的。

💡 关键认知：召回+重排的两层架构，本质是在速度和精度之间找平衡——召回负责"快速缩小范围"（从10万→10），重排负责"精准命中"（从10→3）。这种"粗筛+精排"的思路在搜索引擎、推荐系统中也是标配。

七、环节⑤：生成（Generation）

一句话：把精选的3个片段+用户问题打包发给LLM，让它基于"证据"回答。

📐 生成阶段: 问题+Top3片段 → Prompt → LLM精准回答

操作流程

用户问题："这个视频是哪个账号做的？"
    +
Top 3 片段：
├─ 片段1："《RAG工作机制详解》由马克的技术工作坊创作..."
├─ 片段2："马克的技术工作坊是B站知名技术科普账号..."
└─ 片段3："这个系列视频讲解RAG系统构建..."
    ↓
组成Prompt发给LLM：
"根据以下3个知识片段，回答用户问题：[片段1][片段2][片段3]
 用户问题：这个视频是哪个账号做的？"
    ↓
LLM回答："马克的技术工作坊"  ✅ 准确！有据可依！

生成的优势

维度 纯LLM（无RAG） RAG+LLM
准确性 可能幻觉 基于文档，有据可依
时效性 知识截止 实时检索最新
成本 全文档Context→Token爆炸 只3个片段→Token极少
可追溯 不知道依据是什么 可以标注出处

💡 关键认知：生成阶段的关键技巧是在Prompt中限制LLM只基于提供的片段回答，不要自由发挥。这通过Prompt Engineering实现——加一句"仅基于以下知识片段回答，如果片段中没有相关信息就说不知道"。

八、完整流程端到端总览

┌─────────┐    ┌─────────┐    ┌────────────────┐

│ 网页/TXT │    │ 300-500T │    │ 存入向量数据库    │
└─────────┘    └─────────┘    └────────────────┘
                                        ↓
                               知识库构建完毕 ✅

【提问后·在线问答】
┌─────────┐    ┌──────────┐    ┌──────────┐    ┌────────────┐
│ 用户提问  │ → │ ③召回     │ → │ ④重排     │ → │ ⑤生成       │
│"产品价格?"│    │ 向量搜索   │    │ Cross-Enc │    │ 问题+Top3   │

└─────────┘    └──────────┘    └──────────┘    └────────────┘

每一环节都有其必要性： - 分片让文档可处理 → 索引让文档可搜索 → 召回缩小范围 → 重排提高精度 → 生成输出答案 - 顺序不能颠倒，每步的输出是下步的输入
