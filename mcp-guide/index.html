📖 内容综述

MCP（Model Context Protocol）被称为"AI世界的USB-C接口"——它解决了一个实际痛点：每个AI平台调用工具的方式都不一样，开发者要为每个平台重复造轮子。MCP用一套标准协议统一了工具的注册和调用，写一次Server，到处可用。

这份笔记整合了三部曲的全部内容：基础篇教你"MCP是什么、怎么用"——从安装Cline到配置Server到使用第三方工具；进阶篇教你"MCP怎么建、底层怎么通信"——用FastMCP写自己的Server、用JSON-RPC 2.0理解底层协议；番外篇揭秘"大模型和Host到底怎么交互"——通过抓包发现Cline的4.8万字System Prompt、XML格式工具调用、以及MCP和Function Calling的真正关系。

三部曲的递进逻辑：会用 → 会建 → 理解本质。

第一部分：基础篇——MCP是什么、怎么用

📐 MCP的USB-C比喻: 从碎片化到统一标准

一、MCP简要介绍

一句话：MCP是Anthropic推出的标准化协议，让大模型能通过统一接口使用各种外部工具。

USB-C比喻

MCP之前 = 充电器碎片化时代
├─ ChatGPT用一套工具格式
├─ Claude用另一套
└─ Gemini又不一样
→ 开发者为每个平台写不同的集成代码

MCP之后 = USB-C统一时代
└─ 所有平台用同一个MCP协议
→ 写一次Server，到处可用

三个核心角色

角色 是什么 类比 例子
MCP Host 支持MCP的AI应用 手机 Claude Desktop、Cursor、Cline
MCP Server 提供工具的程序（通常本地运行） 充电器 天气服务、数据库服务
Tool Server内部的具体功能 充电线 get_forecast、get_alerts

💡 关键认知：MCP Server不是"远程服务器"——它通常就是一个本地运行的Python/Node脚本。叫"Server"只是因为它在MCP协议中扮演"提供服务"的角色。

📐 MCP三层架构: Host→Server→Tool

二、配置MCP Server

JSON配置文件格式

{
  "mcpServers": {
    "weather": {
      "disabled": false,
      "timeout": 60,
      "command": "python",
      "args": ["weather.py"],
      "transportType": "stdio"
    }
  }
}
字段 含义
command 启动程序（python/uvx/node/cmd）
args 启动参数（脚本路径）
transportType 传输方式，通常为stdio
timeout 超时时间（秒）

📐 MCP完整交互时序

三、MCP交互流程

阶段一：初始化握手

① Cline读取JSON配置，启动MCP Server进程
② 握手：Cline说"你好"，Server回"我是weather"
③ 工具发现：Cline问"你有什么工具？"
④ Server返回工具列表（名称+描述+参数格式）

阶段二：工具调用

① 用户："明天纽约天气怎样？"
② Cline把问题+可用工具列表发给大模型
③ 大模型判断需要调用get_forecast
④ Cline向MCP Server发调用请求
⑤ Server执行函数，调用真实天气API
⑥ 结果返回给Cline → 转发给大模型
⑦ 大模型总结数据，生成自然语言回答

📐 两种安装MCP Server的方式

uvx方式（Python生态）

"fetch": {

  "args": ["mcp-server-fetch"]

第二部分：进阶篇——自己写Server、理解协议

📐 用FastMCP写Server: 只需3步

五、用FastMCP写MCP Server

最小示例

from mcp.server.fastmcp import FastMCP
import httpx

mcp = FastMCP("weather", log_level="ERROR")

@mcp.tool()
async def get_forecast(latitude: float, longitude: float) -> str:
    """根据经纬度获取天气预报"""
    async with httpx.AsyncClient() as client:
        resp = await client.get(f"https://api.weather.gov/points/{latitude},{longitude}")
        return resp.text

if __name__ == "__main__":
    mcp.run(transport='stdio')

FastMCP三大装饰器

装饰器 用途 例子
@mcp.tool() 注册可调用工具 天气查询、文件操作
@mcp.resource() 定义数据资源 配置文件、用户档案
@mcp.prompt() 构建提示模板 多轮对话模板

💡 关键认知：@mcp.tool()的三个魔法——自动从函数签名提取参数类型、从docstring提取工具描述、自动注册到Server。docstring的质量直接决定模型调用的准确率。

📐 MCP底层: JSON-RPC 2.0 三步握手

六、底层协议：JSON-RPC 2.0

MCP底层通信用的是JSON-RPC 2.0标准，通过stdio（标准输入输出）传输。

三步通信流程

Step 1：initialize（握手）

→ {"method":"initialize","params":{"clientInfo":{"name":"Cline"}},"jsonrpc":"2.0","id":0}
← {"result":{"serverInfo":{"name":"weather"}},"jsonrpc":"2.0","id":0}

Step 2：tools/list（工具发现）

→ {"method":"tools/list","jsonrpc":"2.0","id":1}
← {"result":{"tools":[{"name":"get_forecast","description":"获取天气预报",
    "inputSchema":{"properties":{"latitude":{"type":"number"}}}}]},"jsonrpc":"2.0","id":1}

Step 3：tools/call（工具调用）

→ {"method":"tools/call","params":{"name":"get_forecast",
    "arguments":{"latitude":40.71,"longitude":-74.0}},"jsonrpc":"2.0","id":4}
← {"result":{"content":[{"type":"text","text":"纽约天气：晴..."}]},"jsonrpc":"2.0","id":4}

💡 关键认知：掌握JSON-RPC协议后，你可以绕过任何Host，直接在终端输入JSON指令与MCP Server通信。这证明了MCP协议的独立性和标准化。

第三部分：番外篇——抓包揭秘

📐 抓包揭秘: 4万字System Prompt的真相

七、抓包分析Cline与大模型的交互

抓包方法

在Cline和模型API之间搭一个代理服务器（FastAPI），记录所有请求和响应到日志文件。

三大发现

发现一：4.8万字的System Prompt

Cline发给模型的系统提示词约4.8万字符，包含： - 所有内置工具的XML格式调用规范 - 所有已加载MCP工具的描述和参数 - 执行规则、错误处理、安全约束 - 要求使用<thinking>标签思考、<attempt_completion>结束任务

发现二：XML格式工具调用（不是Function Calling！）

<use_mcp_tool>
<server_name>weather</server_name>
<tool_name>get_forecast</tool_name>
<arguments>{"latitude": 40.7, "longitude": -74.0}</arguments>
</use_mcp_tool>

Cline用Prompt Engineering（提示工程）而非原生Function Calling来实现工具调用。

发现三：Agent的ReAct循环在HTTP层面的体现

第1次POST请求：System Prompt + 用户问题
  → 模型返回：<thinking>+<use_mcp_tool>
第2次POST请求：完整历史 + Observation
  → 模型返回：<write_to_file>
第3次POST请求：更新的历史 + 新Observation
  → 模型返回：<attempt_completion>（任务完成）

每一轮都是完整的新HTTP请求（包含完整历史），Token消耗随对话轮数线性增长。

📐 MCP vs Function Calling 对比

💡 关键认知：番外篇抓包揭示了一个重要发现——Cline的System Prompt有4万多字！其中大部分是MCP工具的描述文本。这意味着每一次LLM调用都要把所有工具描述发一遍，Token消耗巨大。这就是为什么工具描述要写得简洁——因为它会被反复注入到每次请求中。

八、MCP vs Function Calling：互补非替代

┌──────────────────────────────┐
│  大模型 (Claude/GPT/DeepSeek)  │
└──────────────┬───────────────┘
               ↓ Host自己决定格式（XML/FC/其他）
┌──────────────────────────────┐
│  MCP Host (Cline/Cursor)      │
└──────────────┬───────────────┘
               ↓ MCP协议（JSON-RPC 2.0）← 这里才是MCP
┌──────────────────────────────┐
│  MCP Server (Weather/DB/...)  │
└──────────────────────────────┘
维度 MCP Function Calling
作用域 Host ↔ Server Host ↔ Model
标准化 ✅ 统一标准 ❌ 各家不同
解决问题 工具集成碎片化 模型对工具的理解
比喻 手脚（执行能力） 大脑（决策能力）
关系 互补，不是替代 互补，不是替代

📐 MCP vs Function Calling 层级关系

💡 关键认知：MCP不规定模型怎么调用工具——Cline用XML、Cherry Studio用Function Calling、其他Host可以用任何格式。MCP只管"Host和Server之间怎么通信"这一段。理解这一点，你就理解了MCP的真正定位。
