Agent Skill 从使用到原理 学习笔记

课程：B站 · 马克的技术工作坊 B站视频：BV1cGigBQE6n（单视频，约 17 分 42 秒） 讲师：马克 适合人群：用过 Claude/CodeBuddy 但还没搞懂 Skill 是什么的开发者、想区分 Skill 与 MCP 的 Agent 工程师

内容综述

这份笔记把 Agent Skill 这个最近 Anthropic 推出的机制从外到内讲清楚，分四个递进的模块：

第一部分「Skill 到底是个啥」用一个最朴素的比喻给 Skill 下定义——Skill 就是一份给 AI 的岗位 SOP，是一个文件夹，里面装着教 AI 怎么干活的说明书。和 Prompt（一次性指令）、RAG（查资料）、MCP（接外部工具）相比，Skill 是程序化知识（Procedural Knowledge）——告诉模型"该怎么做"。

第二部分「Skill 怎么用」用马克老师在视频中的"会议总结助手"案例，演示 Skill 的标准目录结构（SKILL.md + scripts/ + references/ + assets/）和 frontmatter 写法，以及用户视角下 Skill 是如何被自动触发的。

第三部分「渐进式披露的三层秘密」是 Skill 最精妙的设计。Skill 像一本书：你不会把整本书摊开，而是先看封面（元数据）→ 翻目录（SKILL.md）→ 跳到具体章节（资源文件）。这套三层加载机制让 20 个 Skill 也只占 2000 token，而不是 60000 token。

第四部分「Skill vs MCP，到底用哪个」给出一句话区分："MCP 连数据，Skill 教方法"——MCP 解决"AI 怎么连外部世界"，Skill 解决"AI 怎么干这件事"。两者不是替代关系，而是组合关系。

一、Skill 到底是个啥

📐 图1：Skill 是什么——给 AI 的岗位 SOP

1.1 一句话定义

一句话：Agent Skill 就是一份给 AI 的岗位 SOP——一个文件夹，里面装着教 AI 干某类活的说明书。

马克老师在视频里反复强调一个比喻：Skill 是大模型可以随时翻阅的说明文档。你雇了一个新员工，给他一本"会议纪要怎么写"的工作手册，他需要时翻一翻，不需要时就放在书架上——Skill 就是这本手册。

💡 关键认知：Skill 不是 Plugin，也不是 RAG 文档。它是写给模型自己看的操作指南，模型根据当前任务自主决定要不要"翻这本手册"。

1.2 Skill 的标准目录结构

每个 Skill 就是一个文件夹，长这样：

my-skill/
├── SKILL.md          ← 必须，YAML frontmatter + Markdown 指令
├── scripts/          ← 可选，可执行脚本（python/shell等）
├── references/       ← 可选，参考文档（详细规范、API手册等）
└── assets/           ← 可选，模板/样式/图片等静态资源
部分 作用 类比
SKILL.md 入口文件，模型一定会读 工作手册的封面+目录
scripts/ 确定性代码，由模型调用 工具箱里的螺丝刀
references/ 详细规则，按需查阅 工具箱里的厚厚一本说明书
assets/ 模板、图片、样式 工具箱里的标准件库

💡 关键认知：scripts 里的代码不进入对话上下文——AI 直接执行脚本，只把输出结果带回来。这让 Skill 既能调用 100% 确定的工具（代码），又不浪费宝贵的上下文窗口。

📐 图2：SKILL.md 的标准结构（YAML Frontmatter + Markdown 主体）

1.3 SKILL.md 的 frontmatter 格式

SKILL.md 文件开头必须有一段 YAML 配置，这是 Anthropic 官方的开放规范：

---
name: pdf-toolkit
description: Use this skill whenever the user wants to do anything with PDF files, including reading, extracting text/tables, merging, splitting, rotating, watermarking, encrypting.
license: Apache-2.0
allowed-tools: Read, Write, Bash
---

# 这里开始是 Markdown 主体
当用户提到 PDF 处理时...

核心字段：

字段 是否必须 说明
name ✅ 必须 Skill 唯一标识符（小写，连字符分隔）
description ✅ 必须 何时使用、做什么——这段话是 LLM 决定要不要加载该 Skill 的唯一依据
license 可选 开源协议（Apache-2.0/MIT 等）
allowed-tools 可选 限制该 Skill 可以使用的工具集合
compatibility 可选 兼容性声明

💡 关键认知：description 字段不是写给人看的，是写给模型看的。它必须包含触发条件（"当用户……"）和能力概述（"做什么"）。这两件事写不清楚，Skill 就不会被激活。

二、Skill 怎么用

📐 图8：用户视角的 Skill 调用流——从一句话到结果

2.1 用户视角的工作流

马克老师在视频里用了一个"会议总结助手"的例子来演示。整个流程对用户是透明的：

用户："帮我把这段会议录音整理成纪要"
    │
    ▼
Claude 扫描所有已注册 Skill 的 description
    │
    ▼
匹配到「meeting-summary」Skill → 加载 SKILL.md
    │
    ▼
按 SKILL.md 的步骤：转写→提取要点→生成结构化纪要
    │
    ▼
返回纪要给用户

用户什么都没做——没有输入 /skill xxx，没有点按钮，只是说了一句话。Skill 是模型自主激活的。

2.2 三种递进的使用场景

马克老师把 Skill 的复杂度分成了三档，从基础到高级：

场景 复杂度 需要的文件 例子
基础场景 ⭐ 只要 SKILL.md 会议纪要格式规范、文档写作风格
Reference 场景 ⭐⭐ SKILL.md + references/ API 文档查询、复杂格式规范（如 OOXML）
Script 场景 ⭐⭐⭐ SKILL.md + scripts/ PDF 处理、Excel 计算、需要确定性结果的任务

💡 关键认知：能用文字解释的就别写代码，能写代码的就别堆文字。简单工作流（会议纪要怎么排版）用纯指令；需要 100% 准确的操作（解 PDF 表格）必须用脚本。

2.3 Anthropic 官方 Skill 示例

Anthropic 在 github.com/anthropics/skills 开源了一批官方 Skill，覆盖最常见的办公场景：

Skill 解决什么问题 典型用法
pdf PDF 读取、合并、拆分、加水印 "把这两个 PDF 合并"
xlsx Excel 创建、读取、公式计算 "把这份 CSV 整理成带图表的 Excel"
pptx PPT 创建、模板套用 "做一份产品介绍 PPT"
docx Word 文档处理 "生成一份带表格的报告"

每个 Skill 都遵循同样的三件套：SKILL.md + scripts/（关键 Python 库封装）+ references/（详细规范）。

三、渐进式披露——Skill 的灵魂

3.1 为什么要有渐进式披露

先算一笔账：假设你有 20 个 Skill，每个 SKILL.md 平均 3000 tokens。

加载方式 Token 消耗 后果
❌ 一次性全加载 20 × 3000 = 60,000 tokens 直接爆上下文窗口
✅ 渐进式披露 启动时 20 × 100 = 2,000 tokens 爽爽用

差距 30 倍。这就是为什么 Skill 必须分层加载。

💡 关键认知：上下文窗口是 AI 的"工作记忆"，每多塞一个 token 都是钱。Skill 设计的核心矛盾就是——让 AI 知道有哪些能力可用，但又不能让所有能力都常驻在脑子里。渐进式披露就是这个矛盾的优雅解。

📐 图3：渐进式披露三层机制——Skill 像一本书

3.2 三层披露机制

马克老师用翻书做比喻：你不会把整本书摊开摆桌上，而是先看封面 → 翻目录 → 跳到具体章节。

层级 内容 Token 量 何时加载 类比
Level 1 元数据 name + description ~100 tokens 启动时全部加载 📕 书的封面
Level 2 指令 SKILL.md 主文件 <5000 tokens Skill 被激活时 📑 书的目录
Level 3 资源 references/scripts/assets 任意大 模型按需 Read 📖 具体章节

3.3 每一层的工作机制

Level 1（元数据层）：所有 Skill 的 name + description 在 Claude 启动时全部装进系统提示。这一层只占百来 token，让模型知道"我有哪些招"。

Level 2（指令层）：当用户的请求匹配到某个 Skill 的 description 时，Claude 才会把这个 Skill 完整的 SKILL.md 读进上下文。这一层告诉模型"这招怎么使"。

Level 3（资源层）：SKILL.md 里会写"详细规范见 references/format.md"或"调用 scripts/extract.py"。模型按需用 Read 工具或 Bash 工具去取，只有真正用到的部分才进上下文。

💡 关键认知：scripts 里的代码本体永远不进上下文——AI 只看到调用结果。这是 Skill 与 RAG 的本质区别：RAG 是把内容塞进上下文让模型理解，Skill 是把代码留在外面让模型执行。

四、Skill vs MCP 深度对比

4.1 一句话区分

💡 关键认知：MCP 连数据，Skill 教方法。 - MCP 解决"AI 怎么连接外部世界"——它是协议，是工具，是数据通道 - Skill 解决"AI 怎么干这件事"——它是知识，是 SOP，是程序化指南

📐 图6：Skill vs MCP——一图看清本质区别

4.2 六维对比

对比维度 Agent Skill MCP（Model Context Protocol）
本质 文件包（指令+脚本+资源） 协议+服务器（提供工具/资源/Prompt）
加载方式 模型自主选择，按需读取 客户端主动连接，常驻可用
上下文占用 三层渐进，~100 token 起 工具列表常驻系统提示
解决问题 "怎么做"（程序化知识） "连什么"（外部数据/工具）
典型场景 PDF 处理、报告生成、风格规范 数据库查询、API 调用、文件系统
谁来定义 文件作者（写 SKILL.md） 服务器作者（实现 MCP Server）

4.3 何时用 Skill、何时用 MCP

任务类型 推荐方案 原因
处理 PDF/Excel/Word ✅ Skill 工作流复杂、有标准 SOP
查询公司数据库 ✅ MCP 需要实时连接外部系统
制作品牌一致的 PPT ✅ Skill 涉及大量风格规范和模板
调用 GitHub/Jira API ✅ MCP 标准 API，工具属性强
写一份合规报告 ✅ Skill + MCP 组合 MCP 取数据，Skill 套规范

💡 关键认知：两者不是二选一，是组合拳。最强大的 Agent 是"用 MCP 拿到数据 → 用 Skill 加工成产品"。比如：MCP 从财务系统取销售数据，Skill 把数据变成符合公司风格的季度报告。

4.4 决策流程

我要解决什么问题？

    ├─ 需要连外部系统/实时数据？ ──→ MCP

五、关键洞察总结

把全片浓缩成 6 条最值得记住的认知：

💡 认知 1：Skill 是程序化知识，不是 Plugin、不是 RAG 文档。它是写给模型自己看的"工作手册"。

💡 认知 2：Skill 的核心矛盾是"全装太占上下文，不装模型不知道"——渐进式披露就是这个矛盾的解。

💡 认知 3：scripts 里的代码不进上下文，模型只看输出。这让 Skill 兼具"代码的确定性"和"自然语言的灵活性"。

💡 认知 4：description 字段是 Skill 的"招牌"——写不清楚，模型就不会激活它。一定要包含触发条件和能力概述。

💡 认知 5：MCP 连数据，Skill 教方法——一句话区分两者。最强 Agent 是 Skill + MCP 组合拳。

💡 认知 6：Skill 是 Anthropic 推出的开放规范（不是 Claude 独占），未来会和 MCP 一样成为 AI Agent 的基础设施。
