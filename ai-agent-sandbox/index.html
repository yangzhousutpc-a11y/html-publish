一、沙箱是什么

1.1 一句话定义

沙箱（Sandbox）就是一个给代码画的"围栏"——程序在围栏里怎么折腾都行，但翻不出围栏、碰不到宿主机、也影响不到旁边的其他围栏。

📐 沙箱核心概念——围栏隔离

1.2 为什么AI Agent特别需要沙箱

视频用一个"翻车现场"开场：AI Agent执行了sudo rm -rf /。如果没有沙箱，宿主机直接被清空；有了沙箱，代码只在围栏里跑，删完一键重置，宿主机毫发无损。

风险类型说明例子
代码来源不可信LLM生成的代码可能是幻觉、注入或攻击载荷Prompt注入让Agent执行恶意Shell命令
自主性风险Agent会执行代码、访问网络、连数据库、删文件一次提示词注入=不可挽回的后果
安全边界失效用户习惯性批准权限弹窗Anthropic统计：93%的权限弹窗被盲目批准
💡 关键认知：沙箱不是某一个具体技术，而是一种安全理念——"默认不信任，隔离后再执行"。虚拟机、容器、MicroVM都是沙箱的不同实现形态。

1.3 沙箱的本质——从技术谱系看

实现形态隔离原理隔离强度
chroot/namespaces进程级视图隔离最弱
Docker容器Namespace+Cgroup一般
MicroVM独立Guest内核+KVM强
完整虚拟机硬件虚拟化最强

关键矛盾：隔离越强，开销越大。传统VM安全但启动要几秒到几分钟、内存几百MB到几GB；Docker轻快但共享内核，逃逸风险高。CubeSandbox要解决的，就是在这个"不可能三角"里找第四条路。

二、沙箱 vs 虚拟机 vs 容器

2.1 核心比喻

技术比喻含义
虚拟机（VM）"给你一间全封闭的独立别墅"安全性最高，开销最大
Docker容器"给你的房间装上防盗门和监控，但大家共用一套供水排水系统"性能和兼容性的平衡点，但内核共享有风险
CubeSandbox（MicroVM）"给你一间极速搭建的独立小公寓"独立内核+启动飞快+极低开销

2.2 硬核对比数据

2.3 容器逃逸——最关键的那个"坑"

Docker隔离的本质是"共享内核"，这是很多初学者搞混的地方：

Docker用Namespace做进程隔离、用Cgroup做资源限制——这看起来像"隔离"，但本质是视图隔离（看不到别人），不是安全隔离（无法防止别人攻击你）。Linux内核文档明确说明：Namespace是为了"封装"，不是"安全隔离"。

📐 Docker共享内核 vs CubeSandbox独立内核

真实逃逸案例：CVE-2024-21626（runc容器逃逸）、CVE-2022-0492（cgroup容器逃逸）——每年都有新的容器逃逸漏洞被发现。

💡 关键认知：Docker的Namespace是"视图隔离"（看不到别人），不是"安全隔离"（防不住别人）。Linux内核文档自己都说了Namespace是为"封装"设计的，不是"安全隔离"。一个内核漏洞 = 全部容器沦陷。

4.6 沙箱生命周期管理

创建流程（关键代码）：

async fn create(&self, req: CreateTaskRequest) -> Result<CreateTaskResponse> {
    let config = SandboxConfig::from_bundle(&req.bundle)?;
    let vm = self.resource_pool.acquire(config.template_id).await?;  // 1.资源池获取
    let sandbox = vm.snapshot_clone(SnapshotConfig { ... }).await?;   // 2.快照克隆
    self.network_agent.assign_ip(&sandbox).await?;                   // 3.网络配置
    sandbox.start().await?;                                           // 4.启动
    Ok(CreateTaskResponse { pid: sandbox.pid(), ..Default::default() })
}

销毁流程（关键：快照回收复用）：

async fn delete(&self, req: DeleteRequest) -> Result<DeleteResponse> {
    let sandbox = self.remove_sandbox(&req.id)?;
    sandbox.destroy().await?;                                       // 清理差异页
    self.resource_pool.recycle(sandbox.base_snapshot()).await;       // 回收基础快照
    Ok(DeleteResponse::default())
}

销毁不是真正删除，而是清理差异页；基础快照保留并回收到资源池；下一个请求可立即复用该快照。

五、实测CubeSandbox

5.1 四步跑起来

# 步骤1：克隆并准备环境
git clone https://github.com/TencentCloud/CubeSandbox.git
cd CubeSandbox/dev-env
./prepare_image.sh && ./run_vm.sh && ./login.sh

# 步骤2：一键安装
curl -sL https://cnb.cool/CubeSandbox/CubeSandbox/-/git/raw/master/deploy/one-click/online-install.sh | MIRROR=cn bash

# 步骤3：创建模板
cubemastercli tpl create-from-image \
  --image cube-sandbox-cn.tencentcloudcr.com/cube-sandbox/sandbox-code:latest \
  --writable-layer-size 1G --expose-port 49999 --expose-port 49983 --probe 49999

# 步骤4：用E2B SDK执行代码
export E2B_API_URL="http://127.0.0.1:3000"
export E2B_API_KEY="dummy"
python3 -c "
from e2b_code_interpreter import Sandbox
import os
with Sandbox.create(template=os.environ['CUBE_TEMPLATE_ID']) as sb:
    print(sb.run_code('print(\"Hello from Cube Sandbox!\")'))
"

5.4 当前限制

限制说明
架构支持仅x86_64，ARM暂不支持
硬件要求需KVM硬件虚拟化支持（裸金属或支持嵌套虚拟化的云主机）
GPU直通暂不支持
替代方案无裸金属时可通过WSL 2+QEMU虚拟机体验
💡 关键认知：CubeSandbox的"不可能三角"突破不是魔法，而是4个工程技巧的组合拳：资源池预分配+快照克隆CoW+EPT Lazy Load+全栈锁优化。每个技巧单独看不新鲜，但组合在一起就实现了MicroVM级别的安全+容器级别的性能。
