这期视频回答的是一个让很多 Agent 开发者抓狂的问题：为什么我已经用了最贵的模型、调了上百版提示词，Agent 还是跑不稳？ 花园老师用一个真实的客户故事开篇——朋友的 Agent 任务成功率卡在 70%，最后改动最大的不是模型也不是 Prompt，而是"任务怎么拆、状态怎么管、关键步骤怎么校验、失败之后怎么恢复"。新版上线之后，同样的模型、同样的提示词，成功率拉到了 95% 以上。这一通"不知道叫什么"的工程实践，就是最近爆火的 Harness Engineering。

视频明确分三大段讲：演进、构成、实践。第一段把 Prompt → Context → Harness 三次范式迁移说清楚——它们不是术语流行史，而是 AI 系统从单轮回答走向连续执行所必然要面对的三个阶段性问题。第二段把 Harness 拆成 6 个工程组件（上下文装配、工具接口、执行编排、记忆状态、可观测性、失败恢复），并给出"Agent = Model + Harness"这个被 OpenAI 和 LangChain 反复引用的核心公式。

第三段是全片信息密度最高的部分——讲 OpenAI、Anthropic 是怎么真做的：OpenAI 的 Frontier 团队 3 人 5 个月写了 100 万行代码、0 行手写，人类 100% 时间用来设计 Harness；Anthropic 在《Effective Harnesses for Long-Running Agents》里提出 Initializer + Coding 双 Agent 模式，用 feature_list.json 这种"可继承的工件"破解长程失忆问题。最后用一个让人无法忽视的实证收尾——LangChain 在 Terminal Bench 2.0 上同模型仅换 Harness，通过率从 52.8% 飙到 66.5%、排名从 30 名外冲进 Top 5。视频末尾留了个反方观点：Noam Brown 担心下一代推理模型会把今天的复杂 Harness 冲刷成历史包袱——所以"做 Harness"和"做可被未来继承的 Harness"是两件事。

一、开篇钩子：那个成功率从 70% 飙到 95% 的 Agent 到底改了什么

1.1 一个让花园老师"想了好久才知道叫什么"的故事

一句话：年初有个朋友找花园老师调一个 Agent，最强模型也用了，提示词改了上百版，参数也调了不少，任务成功率就是卡在 70% 上不去。

最后改动最大的地方反而不是模型、不是 Prompt，而是这四件事：

改动维度 具体动作
任务怎么拆 把"端到端一锅炖"拆成"目标 → 信息 → 分析 → 产出 → 校验 → 修正"
状态怎么管 中间结果、长期记忆、当前任务状态分离存储，避免信息相互污染
关键步骤怎么校验 在每个关键节点加独立校验，而不是让模型"自己干活自己打分"
失败之后怎么恢复 不是从头重来，而是带着已经积累的上下文继续

结果：同样的模型、同样的提示词，成功率从 70% → 95%+。

朋友问花园老师"你到底改了什么？"——他当时找不到一个准确的词。直到 Harness Engineering 这个概念越来越火，他才意识到：自己改的这些东西，本质上就是 Harness。

1.2 这期视频的"反向开场白"

💡 关键认知：视频开篇就立了一个反共识——真正决定我们的系统能不能稳定地跑起来，往往不是模型本身，而是模型外面那套运行的系统。 如果你还在"换更强的模型 + 调更精的 Prompt + 加更多的上下文"这个循环里转，那你大概率就缺这第三块拼图。

二、三次范式迁移：Prompt → Context → Harness

2.1 三个阶段对应的不是术语流行，而是三个阶段性问题

花园老师在视频里反复强调一句话：Prompt 解决说什么，Context 解决看什么，Harness 解决怎么干。三者不是替代，而是一层套一层——Harness 在更大的系统层面把 Prompt 和 Context 都包进来。

📐 图 1：三次范式迁移——Prompt → Context → Harness 一层套一层
阶段 解决什么 一句话回答 典型实践
Prompt Engineering 表达问题 模型没听懂你在说什么 Few-shot、CoT、Role Play、Output Format
Context Engineering 信息问题 模型没拿到正确的信息 RAG、Memory、Agent Skills、动态上下文装配
Harness Engineering 执行问题 模型在真实链路里跑不稳 任务拆解、状态管理、独立校验、失败恢复

💡 关键认知：这三个问题是一层一层往外扩张的——Prompt 关注的是字符串、Context 关注的是信息管线、Harness 关注的是整个运行时基础设施。新概念的出现不是因为旧概念过时，而是因为我们要解决的问题变大了。

2.2 Prompt Engineering：塑造一个局部的概率空间

视频原话：

提示词的本质不是命令模型，而是塑造一个局部的概率空间。 提示词解决的是表达的问题，不是信息的问题。

举个例子，"帮我总结这篇文章"和"用三段话总结这篇文章，每段不超过 50 字，第一段是核心结论"——后者更精确，是因为它把模型可能输出的概率分布"压"得更窄。

但只调 Prompt 解决不了一个根本问题：模型未必"知道"。

2.3 Context Engineering：在合适的时机送进合适的信息

典型场景："帮我分析这份需求文档，找出潜在风险，结合历史评审意见，给出改进建议，再生成一版发给产品经理的反馈稿。"

这种任务一句 Prompt 解决不了，必须把 7 类信息有节奏地注入：

当前需求文档 → 历史评审记录 → 相关规范 → 当前目标 → 中间结论 → 输出对象 → 语气要求

Context Engineering 的核心动作：在合适的时机把正确的信息送进去。RAG、Memory、Agent Skills 都属于这个范畴。

💡 关键认知：Prompt 其实只是 Context 的一部分。当 Prompt 静态写死的"系统提示词"装不下全部信息时，就需要一个动态装配信息的工程层——这就是 Context Engineering。

2.4 Harness Engineering：解决"怎么干"的问题

但 Context 也有边界：给了对的信息≠Agent 能跑通整条链路。

举个例子：你给 Agent 全部需求文档+历史评审+规范，让它"实现一个完整功能"。它可能： - ① 一口气干到一半上下文耗尽，留下半成品（One-shotting 失败） - ② 没拿到工具反馈，错了一步还浑然不觉 - ③ 写完代码自己读一遍觉得"OK"就停了，但实际跑不起来

这三类失败和模型聪不聪明、Prompt 写得好不好关系不大——它们都是执行层面的问题：连续多轮怎么不跑偏、状态怎么管、错误怎么发现、失败怎么恢复。

Harness Engineering 关注的就是这一层：模型在真实执行里能不能持续地做对。

三、Harness 是什么：OpenAI 官方定义与那个核心公式

3.1 Harness 不是新词，但 Harness Engineering 是

字面意思：Harness 在英文里是"马具、缰绳"——一套把一头力大无穷但不一定听话的牲口稳稳驾住的装备。

软件工程里它早就有了——Test Harness（测试夹具）：一套用来驱动被测代码、模拟外部依赖、收集结果的脚手架。

新的不是 Harness，而是 Harness Engineering 这个组合词——它把"马具"从一个名词变成了一门专门的工程实践。

3.2 OpenAI 官方定义（视频里被反复引用）

OpenAI 在 2026 年 2 月 11 日发的博客《Harness engineering: leveraging Codex in an agent-first world》里给出了正式定义：

Harness engineering is the design of the runtime layer around the model: tool interfaces, context assembly and compaction, sandboxed command execution, policy enforcement, observability, and failure recovery.

翻译：Harness Engineering 就是设计模型外面的整个运行时层——工具接口、上下文装配与压缩、沙箱命令执行、策略执行、可观测性、失败恢复。

3.3 那个核心公式：Agent = Model + Harness

视频里反复强调的核心公式（来自 OpenAI/LangChain）：

📐 图 2：核心公式 Agent = Model + Harness（车的类比）
            ┌─────────────────────────────────┐
   Agent  = │   Model   +   Harness            │
            └─────────────────────────────────┘
              发动机        车身、方向盘、刹车、仪表盘
              只有少数大厂能炼      所有应用工程师都能调

LangChain 的精彩比喻：

A raw model is not an agent. But it becomes one when a harness gives it things like state, tool execution, and memory. 一个赤裸的模型不是 Agent。当 Harness 给它配上状态、工具、记忆，它才成为 Agent。 模型是 CPU，Harness 才是真正的操作系统。

3.4 Mitchell Hashimoto 的实操定义

HashiCorp 联合创始人、Terraform 作者 Mitchell Hashimoto 给出了一个更接地气的定义（视频也引用了）：

Every time you discover an agent has made a mistake, you take the time to engineer a solution so that it can never make that mistake again. 每次你发现 Agent 犯了一个错，你都花时间工程化一个解决方案，让它永远不再犯这个错。

他还有一句更狠的：

Harness engineering is the practice of engineering a solution even when the core component (the model) is non-deterministic. Harness Engineering 是一种工程实践——即使核心组件（模型）是非确定性的，也要工程化出一个确定性的解决方案。

💡 关键认知：模型本身是非确定性的（同样输入可能给不同输出），但生产系统要求确定性。Harness 的本质就是用工程手段把非确定性的模型驯服成可用的确定性组件——这才是"Engineering"的真正含义。

四、Harness 的六大组件：一份成熟 Harness 必须包含的东西

视频第二大段把 Harness 拆成 6 个工程组件——这正是 OpenAI 官方定义里的 runtime layer 内容。每个组件对应一类典型失败模式。

📐 图 3：Harness 六大组件——成熟 Harness 必须包含的运行时层

4.1 六大组件总览

组件 干什么的 典型失败模式
① 上下文装配与压缩 决定模型每一轮看到什么 注意力涣散、被无关信息淹没
② 工具接口 把外部能力暴露给模型 工具选错、工具太多导致选择困难
③ 执行编排（沙箱） 把任务串成稳定轨道 跑偏、跑飞、不知道自己跑偏了
④ 记忆与状态 任务状态、中间结果、长期记忆分离 长程失忆、状态污染
⑤ 可观测性 / 评估 独立验收 Agent 干得好不好 假象成功（自己干活自己打分）
⑥ 策略执行 / 失败恢复 失败后优雅恢复 系统脆弱、一错就崩

4.2 ① 上下文装配与压缩

核心理念：模型看见什么，比模型会什么更重要。

不能一锅端把所有信息丢给模型，要做分层放置：

信息类型 放置位置 何时注入
不变规则（System Rule） System Prompt 每轮固定
当前任务目标 Working Memory 每轮带
历史中间结论 Compacted Summary 跨轮压缩
工具返回原始数据 Tool Result Slot 用完即清理
长期事实 RAG / 外部记忆 按需召回

💡 关键认知：信息脏比信息少更危险。一段过时的会议纪要可能让 Agent 一直按错的方向努力——压缩不是简单截断，而是有损但保留关键事实的归纳。

4.3 ② 工具接口

反直觉的事实：工具多 ≠ 能力强。

视频引用了 Vercel 的实测数据：

工具数量 任务准确率
15 个工具 80%
2 个工具 100%

工具一多，模型选择困难、token 浪费、调用错误率飙升。

好的工具接口设计： - 渐进式披露（Progressive Disclosure）：在正确时机给正确的工具，而不是一开始把所有工具的 schema 都塞进上下文（这是 Anthropic Agent Skills 的核心思想） - 工具职责单一：每个工具只做一件事，做好一件事 - Schema 简洁：参数描述清楚，返回结构化

4.4 ③ 执行编排：把步骤串成稳定轨道

把"端到端一锅炖"换成有节奏的状态机：

理解目标 → 补信息 → 分析 → 产出 → 检查 → 修正 → 完成
   ↑                                         ↓
   └─── 任何节点失败都能定位回到合适的状态 ────┘

关键设计： - 每个节点有明确的进入条件和输出契约 - 失败时不是从头来，而是回退到最近的稳定状态 - 关键节点强制做校验（不让模型自己说"我做完了"）

4.5 ④ 记忆与状态：信息脏比信息少更危险

三层分离：

记忆类型 生命周期 例子
任务状态 当前会话 当前任务到了哪一步、待办列表
会话中间结果 当前会话 已经查到的事实、已经写的代码片段
长期记忆 跨会话 用户偏好、项目规范、错误教训

💡 关键认知：把这三类混在一个上下文里是大多数 Agent 翻车的根源——上一个任务的脏状态污染下一个任务，长期事实和当前任务夹杂导致模型注意力分散。

4.6 ⑤ 可观测性 / 评估：最容易被低估的一层

最容易踩的坑：让模型自己干活自己打分。

视频里花园老师反复强调：必须有独立的验收系统——独立的 evaluator、自动化测试、真实环境验证，而不是让生成 Agent 写完代码自己读一遍说"OK 没问题"。

Anthropic 后来在《Harness Design for Long-Running Apps》里把这个洞察工程化了——三 Agent 架构：Planner（规划）+ Generator（生成）+ Evaluator（独立评估）。

4.7 ⑥ 策略执行 / 失败恢复

核心理念：失败是常态而非异常。

工程化失败恢复的几个关键点： - 错误信息可机读：让上层 Harness 能从错误里提取重试策略 - 错误沉淀为规则：每犯一次错，把"不能再犯"写进规则库（呼应 Mitchell Hashimoto 的定义） - 优雅降级：核心路径走不通时有备用路径，而不是直接报错

五、OpenAI 实战案例：3 人团队 5 个月 100 万行代码 0 行手写

5.1 这个故事到底有多炸

视频里花园老师讲到这个案例时明显兴奋——这是 Harness Engineering 概念能"出圈"的最重要催化剂。

OpenAI Frontier Product Exploration 团队： - 起点：3 个人（后扩到 7 人）、空 git 仓库 - 完全禁止人工手写一行代码 - 时间：约 5 个月 - 产出：约 100 万行代码、约 1500 个 PR、人均日处理 3.5 个 PR - 整体效率提升约 10 倍

5.2 关键论断

OpenAI 工程师 Ryan Lopopolo 的原话（视频引用了）：

💡 关键认知：Agents aren't hard; the Harness is hard.（Agent 不难，难的是 Harness。）

这句话的潜台词是——Agent 框架谁都能搭起来一个 demo，能跑通的关键全在 Harness 里。

5.3 Symphony：他们造的 Harness 系统

OpenAI 自己开发了一个名为 Symphony 的 Harness，由四层组成：

📐 图 4：OpenAI 的 Symphony 四层架构（3 人团队 5 个月 100 万行 0 手写）
层 名字 干什么 类比
前馈控制层 Guides PRD、架构说明书、代码规范——让 Agent 动手前"想清楚" 蓝图、施工图
反馈控制层 Sensors 自动化 Linter、单元测试、沙箱环境——让 Agent 自我修正 仪表盘、传感器
执行运行时 Runtime Codex App Server（JSON-RPC）、双向通信、线程生命周期管理 操作系统
生命周期管理 GC 自动状态重置——防止 token 膨胀和上下文熵增 垃圾回收

5.4 人类工程师的角色彻底变了

从"实现状态"到"声明期望状态"：

  • 不再亲自写代码
  • 100% 的时间用来设计 Harness
  • 通过声明式 Prompt 告诉 Agent"我想要什么"，让 Symphony 自动跑通整个流程

OpenAI 把这个交互范式叫做 The Ralph Wiggum Loop——人类掌舵、智能体执行。

💡 关键认知：这不是科幻——这就是 OpenAI 内部已经跑了 5 个月的工作模式。如果你今天还把所有时间花在"调 Prompt"上，方向可能错了。未来 Agent 工程师的核心技能不是写 Prompt，而是设计 Harness。

六、Anthropic 实战案例：双 Agent 模式破解长程失忆

6.1 长程任务最让人崩溃的问题

Anthropic 在 2026 年 3 月发的《Effective Harnesses for Long-Running Agents》提出了一个核心痛点：

Agent 在多个上下文窗口之间工作，每个新会话都没有上一会话的记忆——就像每班来的工程师都失忆。

仅靠 compaction（上下文压缩）不够——压缩后的摘要会丢失关键细节和隐性假设。

6.2 两类典型的长程任务失败

失败模式 现象 后果
One-shotting Agent 想一次做完整个 app 跑到一半上下文耗尽，留下半成品
过早宣布完成 后来的 Agent 看到已经做了一些功能 以为完事了，停下来

6.3 Anthropic 的解法：Initializer + Coding 双 Agent

架构图：

📐 图 5：Anthropic 双 Agent 模式——破解长程失忆的"分班接力"
              ┌──────────────────────────────────────┐
              │       Initializer Agent（只跑一次）   │
              │  ┌───────────────────────────────┐   │
              │  │ 创建 init.sh 启动脚本          │   │
              │  │ 创建 claude-progress.txt 日志  │   │
              │  │ 做初始 git commit             │   │
              │  │ 生成 feature_list.json（核心）│   │
              │  └───────────────────────────────┘   │
              └──────────────────────────────────────┘
                              │ 工件传递
                              ▼
              ┌──────────────────────────────────────┐
              │   Coding Agent（每个 session 跑一次） │
              │  1. pwd 确认目录                     │
              │  2. 读 git log + progress 了解上次   │
              │  3. 读 feature_list 挑一个未完成的    │
              │  4. 跑 init.sh 做端到端测试          │
              │  5. 实现 → 测试 → commit → 更新日志  │
              │  6. 留下 clean state（可随时 merge） │
              └──────────────────────────────────────┘

6.4 feature_list.json：核心工件设计

Anthropic 用 claude.ai 克隆案例做实验，把高层提示展开成 200+ 个具体功能点：

{
  "features": [
    {
      "id": "F001",
      "name": "用户登录",
      "acceptance": "用户输入邮箱密码后跳转主页",
      "passes": false
    },
    {
      "id": "F002",
      "name": "对话列表侧边栏",
      "acceptance": "显示历史对话，可点击切换",
      "passes": false
    }
    ...
  ]
}

为什么是 JSON 不是 Markdown： - 模型不容易误改 JSON 的结构 - 严格约束：Agent 只能改 passes 字段，其他字段 readonly

💡 关键认知：这就是 Mitchell Hashimoto 那句话的工程化——每发现 Agent 一种失败模式，就把"不能再犯"沉淀成可机读的规则。feature_list.json 不是文档，是"喂给下一班 Agent 的不可变契约"。

6.5 进阶：三 Agent 架构

后续 Anthropic 又加入了 Evaluator Agent——一个独立的、不做生成只做评估的 Agent，灵感来自 GAN：

Planner（规划）→ Generator（写代码）→ Evaluator（独立验收）
                                       │
                                       └── 不通过 → 反馈给 Generator 重做

为什么需要独立 Evaluator：让生成者自我批判会有偏差——它已经投入了大量"心力"在自己的方案里，倾向于自我合理化。独立评估者比让生成者自我批判要可靠得多。

6.6 Claude Code 自己就是 Harness 范例

Boris Cherny 和 Cat Wu（Claude Code 负责人）的设计哲学：

所有的秘密武器都在模型本身。我们追求的是最薄的那层包装。

他们属于 Big Model 阵营——尽量薄，把判断交给模型。

反例数据（视频里也提到了）： - 在 Terminal Bench 2.0 上，Claude Opus 4.6 用 Claude Code Harness 排第 33 - 换一个不同 Harness，同模型同基准冲到了第 5 - 排名相差 28 位——说明模型对特定 Harness 存在过拟合

💡 关键认知：这是个有意思的悖论——模型越强、Harness 越薄越好；但当下任何模型都不够强到不需要 Harness。这两句话同时成立。所以 Claude Code 选了"尽量薄"作为长期目标，但短期里仍然包含了大量精心设计的 Harness 细节。

三个关键改造：

模块 解决什么 怎么改

八、反方观点：Noam Brown 的"Bitter Lesson"派担忧

8.1 视频末尾的反思段

视频不是"Harness 万能论"——花园老师在最后留了一段反方观点。这一段反而是全片最值得深思的部分。

Noam Brown（OpenAI、o1/o3 推理模型核心研究者）在 Latent Space 访谈里给出了反方观点：

之前投入了大量工程来构建这些 Agentic 系统……结果我们造了推理模型之后，你根本不需要那些复杂的行为了。

别花六个月搭建一个可能六个月后就被淘汰的东西。

8.2 历史类比：GPT-4o 时代的"代偿性 Harness"

在推理模型（o1/o3）出现前，开发者在 GPT-4o 上搭了大量复杂 Agentic 系统： - 路由器（让 GPT-4o 判断要不要 think more） - 编排器（把任务拆成多步推理） - Multi-agent（用多个 Agent 互相辩论）

这些工程的目的，本质上是用工程手段模拟模型的推理能力。

结果 o1/o3 一出来，这些复杂结构都不需要了——模型自己内化了推理过程，外面这层 Harness 反而拖累效果。

8.3 阵营分化：Big Model 派 vs Big Harness 派

阵营 立场 代表
Big Model 派 算力增长终将碾平工程花活，Harness 越薄越好 Noam Brown、Claude Code 团队
Big Harness 派 当下模型不够强，必须靠工程驯服 LangChain、OpenAI Codex 团队、Mitchell Hashimoto

💡 关键认知：这两派的分歧不在于 Harness 当下有没有用——双方都承认当下有效。分歧在于未来：你今天构建的 Harness，是会被下一代模型淘汰的临时支架，还是能被新模型继承的结构化资产？
