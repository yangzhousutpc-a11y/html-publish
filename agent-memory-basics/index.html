📖 内容综述

这门课要解决一个最朴素的痛点：**Agent 像金鱼一样，每开一个新对话就忘光了上次的事**。Andrew 在开场把这种状态叫"金鱼脑 / 记忆碎片综合症"——你跟它说过 100 次"我落款写 Lance"，它每次都还是写 Best regards。这门课就是教你怎么给 Agent 装上**长期记忆**，让它越用越懂你。

第一章「**为什么需要记忆**」回答的是"问题诊断"——为什么单纯加大上下文窗口、或者上 RAG 都解决不了这个问题。课上明确把记忆按**认知心理学的 Tulving 框架**分成三类：**Semantic（语义/事实）**、**Episodic（情景/经历）**、**Procedural（程序/指令）**——这是整门课的骨架。

第二章「**构建记忆管理器**」回答的是"工程基建"——LangGraph 提供 `Store`（跨会话）和 `Checkpointer`（线程内）两套存储，靠 `namespace` 像目录一样组织记忆。`Profile`（覆盖式画像）vs `Collection`（追加式集合）是选型的关键。

第三、四章「**语义记忆 + 记忆操作**」回答的是"记忆的生与死"——`Hot Path`（Agent 自己在对话中调工具记）vs `Background`（独立反思器异步整合），课上反复强调这是**人脑边聊边记 vs 回家路上复盘**的对应。再加上去重、update-in-place、ReflectionExecutor 防抖三个关键工程细节。

第五章「**记忆感知智能体**」是整门课的"哇塞时刻"——把 system prompt 本身当成一条记忆放进 store，让 Agent 根据用户反馈**自己改写自己的 prompt**。不需要 fine-tune、不需要 RLHF，纯靠 prompt 改写就实现了在线学习。

最后用一个完整的**邮件助手 demo** 把三类记忆全部装进去，演示从"金鱼版"到"越用越懂你"的完整进化路径。

第一部分：为什么 Agent 需要记忆

一、Agent 的"金鱼脑"问题

痛点演示：跨 Thread 立刻失忆

一句话：现在的 Agent 默认像鱼缸里的金鱼——切到新对话就忘光，包括你刚刚说过的"我喜欢深色模式"。

Harrison 在 P2 开场做了一个反面演示：

  1. 第 1 个 thread：用户跟 Agent 说 "I prefer dark mode for my UI."（我喜欢深色模式）→ Agent 答 OK，记下了。
  2. 第 2 个 thread（新对话）：用户问 "What UI mode do I prefer?" → Agent 一脸懵："I don't have any information about your UI preferences."

💡 关键认知：很多人以为 LangGraph 的 MemorySaver checkpointer 已经"加了记忆"。但 checkpointer 只在同一个 thread_id 内有效——换 thread 就归零。邮件助手每封新邮件几乎都是新 thread，所以只有跨 thread 的 Store 才能让 Agent 真正"长记性"。

三个看似可行但其实解决不了的伪方案

伪方案 看起来能干嘛 为什么不够
加大上下文窗口 把过去对话全塞进去 100 个用户、3 年聊天记录塞不下；找不准；贵
RAG 检索文档 检索外部知识 RAG 找的是"人类写的文档"，不是"关于这个用户的事实"，namespace 也不按用户隔离
MemorySaver checkpointer 保存对话状态 只在同一 thread 内；跨会话归零

💡 关键认知：Agentic Memory ≠ RAG，也 ≠ 更长的上下文窗口。它是一套独立的工程基建：跨会话、按用户/会话隔离、可读可写可更新。

二、三类记忆：来自认知心理学的框架

📐 图2：三类记忆 — Tulving 框架在 Agent 上的对应

Tulving 三分法

一句话：人脑的记忆分三类——你知道的事实、你经历过的事、你会做的动作；Agent 也照着抄。

这门课最重要的"骨架"就是这张三类记忆对照表（来自心理学家 Endel Tulving 1972 年的工作）：

记忆类型 人类对应 Agent 形态 邮件助手里干啥 LangMem API
Semantic（语义） 你知道"巴黎是法国首都" 关于用户/世界的事实（k-v 或文档） 记住"我落款叫 Lance"、"Alice 是 VIP" manage_memory_tool
Episodic（情景） 你记得"去年生日吃了寿司" 过去发生过的具体例子（few-shot） Triage 历史："这种邮件之前我归到 ignore 了" Store 存 {email, label} 检索
Procedural（程序） 你会骑自行车（不用想步骤） Agent 的操作规则 / system prompt 本身 "怎么写邮件、怎么分类邮件"的 prompt 字符串 create_multi_prompt_optimizer

💡 关键认知：三类记忆不是三个 Store，而是同一个 Store 用不同 namespace 区分。理解 namespace = 文件夹路径，是吃透这门课的关键。

短期 vs 长期记忆的存储分层

📐 图1：Checkpointer（短期）vs Store（长期）的存储分层

一句话：短期记忆（当前对话）放 Checkpointer，长期记忆（跨对话）放 Store——两套东西，别搞混。

                  ┌──────────────────────────────┐
   当前对话 ───▶  │  Checkpointer (短期/thread)  │  ◀── 同 thread_id 才看得到
                  │  保存：messages, intermediate │
                  └──────────────────────────────┘

   跨对话/跨用户   ┌──────────────────────────────┐
   长期事实   ───▶ │  Store (长期/cross-thread)   │  ◀── 按 namespace 组织
                  │  保存：facts, examples,       │
                  │       prompts (procedural)    │
                  └──────────────────────────────┘
维度 Checkpointer Store
作用域 单个 thread 跨 thread / 跨会话
内容 整张图的 state（messages、变量） 用户事实、偏好、历史例子、prompt
实现 MemorySaver / PostgresSaver InMemoryStore / PostgresStore
组织方式 按 thread_id 按 namespace = ("app", "user_id", "type")
存活时长 对话结束（取决于 TTL） 永久（直到主动删除）

第二部分：构建记忆管理器（Memory Manager）

一、LangGraph Store 的核心抽象

namespace = tuple = 目录结构

📐 图3：namespace = 目录路径，组织所有记忆

一句话：把 namespace 想成"文件夹路径"，所有记忆都按 (应用名 / 用户 ID / 记忆类型) 分目录存。

LangMem 的官方 namespace 设计模式（课上引用过）：

模式 例子 用途
按用户 ("memories", "{user_id}") 每个用户记忆隔离
按助手 ("memories", "{assistant_id}") 一个助手跨多用户共享记忆
组织 + 用户 ("memories", "{org_id}", "{user_id}") 组织内可搜，用户间隔离
进一步细分 ("memories", "{user_id}", "manual_memories") 不同种类记忆分开

💡 关键认知：{langgraph_user_id} 这种 {} 占位符是 LangMem 的运行时模板语法——不是 f-string，是 LangMem 自己解析的。运行时它从 RunnableConfig.configurable.langgraph_user_id 自动读取并替换。

Profile vs Collection 选型

一句话：身份证（Profile，固定字段、原地覆盖）vs 朋友圈（Collection，无限追加、按时间累积）。

维度 Profile（档案） Collection（集合）
类比 通讯录详细资料页 微信收藏夹
结构 固定 schema（JSON 字段） 一条条独立记录
写法 覆盖（update-in-place） 追加（append）
适合 用户偏好（写邮件风格、会议时长、VIP 名单） 通用事实、人物背景、对话要点
检索 直接 store.get() 一把取出来 store.search() 语义检索 top-k

💡 关键认知：选型决策很简单——字段固定 + 数量小 → Profile；条目无限 + 不预知 → Collection。课上邮件助手两种都用：写邮件偏好用 Profile，过往邮件要点用 Collection。

二、组装一个最小可用的 Memory Manager

五步代码（课上原版）

from langgraph.store.memory import InMemoryStore
from langgraph.prebuilt import create_react_agent
from langgraph.utils.config import get_store
from langmem import create_manage_memory_tool, create_search_memory_tool

# Step 1: 建一个带嵌入索引的 Store
store = InMemoryStore(
    index={"dims": 1536, "embed": "openai:text-embedding-3-small"}
)

# Step 2: prompt 函数（注意是函数不是字符串！）
def prompt(state):
    store = get_store()
    memories = store.search(("memories",), query=state["messages"][-1].content)
    system_msg = f"""You are a helpful email assistant.

## Memories
<memories>
{memories}
</memories>
"""
    return [{"role": "system", "content": system_msg}, *state["messages"]]

# Step 3: 把 LangMem 的两个工具加进 Agent 工具集
agent = create_react_agent(
    "anthropic:claude-3-5-sonnet-latest",
    prompt=prompt,
    tools=[
        create_manage_memory_tool(namespace=("memories",)),  # 写记忆
        create_search_memory_tool(namespace=("memories",)),  # 查记忆
    ],
    store=store,
)

💡 关键认知：create_react_agent 的 prompt 参数可以是一个函数，每轮调 LLM 前会被调用。"先 store.search 取记忆 → 再拼进 system prompt"——这就是把"长期记忆"接入"短期上下文"的接缝。这个细节是初学者最容易漏掉的。

两个核心工具的分工

工具 谁调用 干啥
manage_memory_tool Agent 自己（在对话过程中） create / update / delete 一条记忆
search_memory_tool Agent 自己（在回答前） 语义检索历史记忆

第三部分：用语义记忆扩展 Agent 工具调用

一、为什么 Agent 工具调用需要 Semantic Memory

两个动机故事（课上原版）

故事 A：避免重复犯错 - 上周：Agent 把 Alice 的工程支持邮件当垃圾邮件忽略掉了。 - 用户骂："以后从 Alice 来的邮件别忽略"。 - 没有长期记忆 → 下周 Alice 再来邮件，Agent 还会再忽略一次（因为它早忘了）。 - 有 Semantic Memory → 这条反馈被存下来，下次 Triage 时检索到，不会再犯。

故事 B：记住用户偏好 - 用户：跟 Harrison 之外的人开 30 分钟会，跟 Harrison 开 1 小时；落款写 "Lance"；不用 "Best regards"，用 "Cheers"。 - 这些偏好不应每次都让用户重新告诉 Agent。 - 写一次 → 存进 Semantic Memory → 永久生效。

💡 关键认知：Semantic Memory 解决的不是"知识缺失"问题（那是 RAG 的事），而是"关于这个用户/这次交互的事实"。它是按用户隔离的、长期累积的、可读可写可更新的。

Alice 邮件追踪案例（课上完整演示）

时间点 发生了什么 Agent 的动作

第四部分：记忆操作——提取、整合与自更新

一、Hot Path vs Background：两种写记忆的范式

📐 图4：Hot Path（边聊边记）vs Background（回家复盘）

核心类比（课上原话）

Hot Path 像你边和朋友聊天边在脑子里默念"这点要记住"。 Background 像你聊完天回家路上一边走一边复盘——后者更深、更整合，但延迟。

两种范式对比

维度 Hot Path（热路径） Background（后台）
触发时机 生成回答的过程中 对话结束 / 空闲时
决策方 主 Agent 自己用工具决定 独立反思 Agent（reflector）
当前轮延迟 高（多调一次工具，~+500ms） 0（不阻塞用户）
处理深度 浅（每轮一次） 深（批量整合）
立即可用 是 否（下一轮才生效）
API 工具 create_manage_memory_tool / create_search_memory_tool create_memory_store_manager / ReflectionExecutor
适合谁 Semantic 的轻量写入 Episodic / Procedural 的整合

💡 关键认知：Hot Path 当前轮慢但下轮就用上；Background 当前轮 0 延迟但下一轮才看到效果。两者不是互斥的——生产里通常 Hot Path 处理"用户明确要记的"，Background 处理"对话隐含的事实抽取与整合"。

二、Background 模式的代码（课上演示）

from langchain.chat_models import init_chat_model
from langgraph.func import entrypoint
from langgraph.store.memory import InMemoryStore
from langmem import ReflectionExecutor, create_memory_store_manager

store = InMemoryStore(index={"dims": 1536, "embed": "openai:text-embedding-3-small"})
llm = init_chat_model("anthropic:claude-3-5-sonnet-latest")

# 创建一个"反思器"
memory_manager = create_memory_store_manager(
    "anthropic:claude-3-5-sonnet-latest",
    namespace=("memories",),
)

@entrypoint(store=store)
async def chat(message: str):
    response = llm.invoke(message)
    to_process = {"messages": [{"role": "user", "content": message}, response]}
    await memory_manager.ainvoke(to_process)   # 异步整合
    return response.content

三、记忆冲突 / 去重 / Update-in-place（课上的"魔法演示"）

📐 图6：Update-in-place — 狗改名的语义去重

故事：狗的名字改了

时间 用户输入 Background reflector 干了什么
第 1 轮 "My dog's name is Fido." 提取出 {subject: user, has_dog: "Fido"}，写入 store。
第 2 轮 "Actually I renamed my dog to Buddy." 不会新增一条 has_dog: Buddy；而是根据语义检索找到旧记录的 ID，update-in-place 改成 Buddy。

实现关键参数：

memory_manager = create_memory_manager(
    "anthropic:claude-3-5-sonnet-latest",
    schemas=[Fact],
    instructions="Extract important facts about the user...",
    enable_inserts=True,   # 允许插入新条
    enable_updates=True,   # 允许修改/合并已存在的条
)

LangMem 内部的 reconciliation（调和）流程： 1. 用 embedding 查找语义相似的旧记忆。 2. 让 LLM 决定 "新增 / 更新 / 忽略"。 3. 写入 store。

💡 关键认知：去重不是 hash 去重，是语义去重。"我有只狗" 和 "我养了一只狗" 会被识别成同一条事实——这要求每次写入都过一次 LLM 判断，所以 Background 模式更适合做这种"重活"。

四、ReflectionExecutor：防抖 + 队列

📐 图5：ReflectionExecutor 的 Debounce 机制

为什么需要 Debounce

一句话：用户连发 5 条消息，不应触发 5 次后台反思——浪费钱、还可能矛盾。

from langmem import ReflectionExecutor, create_memory_store_manager

memory_manager = create_memory_store_manager(...)
executor = ReflectionExecutor(reflector=memory_manager)

future = executor.submit_task(
    thread_id="user_123",
    conversation_history=history,
    after_seconds=30,   # ← 防抖：30s 内有新消息就重置计时
)

核心机制（4 步）

1. submit_task 进入优先级队列
       ↓
2. 等待 after_seconds（默认 30s）
       ↓
3. 等待期间同 thread_id 又来新任务？
   ├─ 是 → 取消前一个，用新的重新计时
   └─ 否 → 继续等
       ↓
4. 最终一次对话只跑一次反思，整合所有上下文

💡 关键认知：原来生产环境的记忆系统不是每条消息都立刻反思一次，而是用 debounce 攒一批再整合——这就是为什么 Background 又不阻塞、又质量高的关键。debounce 单位是 thread_id，多用户并发不会互相干扰。

第五部分：记忆感知智能体（Memory-Aware Agent）

一、Procedural Memory：让 Agent 自己改自己的 Prompt

📐 图7：Procedural Memory — 把 prompt 当数据，让 Agent 自己改自己

这一节最大的"哇塞时刻"

"我们一直把 system prompt 当成开发者写死的东西。但 system prompt 本身也可以是一种可学习、可更新的记忆——让 Agent 收集用户反馈，自己改写自己的 prompt。"

Procedural Memory 的实现路径

第一步：初始 prompt 不写死，从 store 里拉

prompt_instructions = {
    "ignore":  "Ignore promotional emails and spam.",
    "notify":  "Notify me for urgent issues regarding service downtime.",
    "respond": "Respond to all customer support inquiries.",
    "agent":   "You are a helpful email assistant. Write concise and professional emails."
}

def create_prompt(state, config, store):
    user_id = config["configurable"]["langgraph_user_id"]
    namespace = (user_id,)
    result = store.get(namespace, "agent_instructions")
    prompt = result.value["prompt"] if result else prompt_instructions["agent"]
    return [{"role": "system", "content": system_template.format(instructions=prompt)},
            *state["messages"]]

第二步：用 create_multi_prompt_optimizer 根据反馈自动改写

from langmem import create_multi_prompt_optimizer

optimizer = create_multi_prompt_optimizer(
    "anthropic:claude-3-5-sonnet-latest",
    kind="prompt_memory",
)

prompts = [
    {
        "name": "main_agent",
        "prompt": store.get(("lance",), "agent_instructions").value["prompt"],
        "update_instructions": "Keep the instructions concise and focused on key behaviors.",
        "when_to_update": "Update when there is feedback on how the agent should write emails or schedule events.",
    },
    {
        "name": "triage_ignore",
        "prompt": store.get(("lance",), "triage_ignore").value["prompt"],
        "update_instructions": "Keep the instructions short and to the point.",
        "when_to_update": "Update whenever there is feedback on which emails should be ignored.",
    },
    # ... triage_notify, triage_respond
]

# trajectories = (前一次的 messages, 用户反馈)
conversations = [
    (response["messages"], "Always sign your emails 'Best regards, Lance'")
]

updated = optimizer.invoke({"trajectories": conversations, "prompts": prompts})

for new_p, old_p in zip(updated, prompts):
    if new_p["prompt"] != old_p["prompt"]:
        store.put(("lance",), f"{old_p['name']}_instructions",
                  {"prompt": new_p["prompt"]})

💡 关键认知：update_instructions 和 when_to_update 是元规则——这两个字段直接喂给改写器作为 meta-prompt，让它知道"什么时候才改、改成什么风格"。这是从"规则编程"到"规则改写规则"的跃迁。

二、两个反馈案例（课上完整演示）

案例 A：邮件签名

步骤 内容
反馈 "Always sign your emails as Lance"
优化器判定 动 main_agent prompt（因为 when_to_update 提到"how to write emails"）
改写后的 prompt "You are a helpful email assistant. ... When sending emails, always sign them as 'Lance'."
验证 下一封邮件自动落款 Lance

案例 B：忽略某人邮件

步骤 内容
错误 Agent 错误回复了 Alice Jones 的无关邮件
反馈 "Ignore any emails from Alice Jones."
优化器判定 动 triage_ignore prompt（when_to_update 命中"which emails should be ignored"）
改写后的 prompt triage_ignore 增加 "Ignore all emails from Alice Jones."
验证 下次 Alice Jones 来邮件直接被 triage 路由到 ignore

💡 关键认知：整个过程没改一行代码，只改 store。这就是"把 prompt 当数据"的威力——Agent 进化不再需要重新部署，只需要写一条记忆。

from langgraph.graph import StateGraph, START
from langgraph.prebuilt import create_react_agent
from langgraph.store.memory import InMemoryStore

store = InMemoryStore(index={"embed": "openai:text-embedding-3-small"})

# Response Agent
response_agent = create_react_agent(
    "anthropic:claude-3-5-sonnet-latest",
    tools=[write_email, check_calendar_availability, schedule_meeting,
           manage_memory_tool, search_memory_tool],
    prompt=create_prompt,   # 内部从 store 拉 procedural memory
    store=store,
)

# 整张图
email_agent = StateGraph(State)
email_agent.add_node("triage_router", triage_router)
email_agent.add_node("response_agent", response_agent)
email_agent.add_edge(START, "triage_router")
email_agent = email_agent.compile(store=store)

💡 关键认知：triage_router 和 response_agent 用同一个 store 但读不同 namespace——这就是为什么 namespace 设计对了，整套体系才优雅： - (user_id, "agent_instructions") → main_agent prompt - (user_id, "triage_ignore") → triage 忽略规则 - ("email_assistant", user_id, "collection") → semantic facts - ("email_assistant", user_id, "examples") → episodic few-shot

第六部分：课程总结

Harrison 的收尾三点

📐 图9：从金鱼脑到完整 Memory-Aware Agent 的进化路径

1. 记忆是让 Agent "越用越懂你"的关键基建

"记忆是让 Agent 从'每次都从零开始'，变成'越用越懂你'的关键基建。"

整门课的进化路径就是把一个金鱼版邮件助手逐层升级：

baseline 助手（金鱼脑）
   ↓ 加 Semantic Memory（事实/偏好）
能记住"我落款写 Lance"
   ↓ 加 Episodic Memory（few-shot 例子）
Triage 越用越准
   ↓ 加 Procedural Memory（prompt 自我改写）
Agent 自己根据反馈进化
   ↓
完整 Memory-Aware Agent

2. 推荐生态工具

工具 用途 地址

LangSmith 追踪、评测、调试 Agent smith.langchain.com

3. Andrew 的下一步建议

  • 把课程仓库 fork 下来跑通邮件助手；
  • 替换成自己的真实场景（客服、研究、代码 Agent）；
  • 关键：先想清楚"该记什么、什么时候记"，再选 hot / background——别上来就铺三类记忆。

三个最值得带走的"恍然大悟"

  1. Procedural Memory 就是把 prompt 字符串本身当数据 在 P6 才会真正打通。在此之前学生大多以为 prompt 是开发期产物。一旦想通"system prompt 也可以是记忆"，整个 Agent 自我进化的逻辑就通了。

  2. Hot Path 慢，Background "当前轮 0 延迟"，但 Background 的"快"是用"下一轮才见效"换来的 ReflectionExecutor + debounce 是这门课唯一稍微"反直觉"的工程细节。理解了这个 tradeoff，才能在生产里做出正确的选择。

  3. 三类记忆其实是同一个 Store 不同 namespace P6 收束时才会意识到。所以namespace 设计是吃透整门课的关键——它不只是命名，而是数据隔离、访问控制、记忆生命周期的总纲。
