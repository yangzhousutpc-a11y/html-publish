📖 内容综述

这门课的核心信息只有一句话：好的Prompt不是试出来的，是工程化设计出来的。 吴恩达和OpenAI的Isa Fulford用9集课讲清了一件事——掌握两大原则（清晰具体+给模型思考时间），然后通过迭代循环不断打磨，你就能让同一个模型的表现产生天壤之别。

课程的递进逻辑很清晰：先学两大原则打基础，再用迭代开发掌握方法论，然后在四大应用（摘要/推理/转换/扩展）中实战练手，最后升级到多轮对话的聊天机器人。从单次Prompt到完整的对话系统，一步步递进。

一、两大原则：Prompt设计的基本功

📐 课程核心框架: 两大原则 + 迭代 + 四大应用

原则一：编写清晰、具体的指令

一句话：清晰 ≠ 简短。清晰意味着让模型毫无歧义地理解你要什么。

📐 两大原则 + 四大应用递进

📐 清晰具体指令的四大策略

四大策略

策略1：使用分隔符

用 ` ` `、"""、<>、### 等符号分隔指令和待处理文本：

请总结以下用三重引号包裹的文本：
"""
这里是需要总结的长文本内容...
"""

为什么重要？ 防止"提示注入"——如果不用分隔符，用户输入的文本可能包含"忘记之前的指令"这样的内容，模型会被误导。

策略2：要求结构化输出

明确要求JSON、HTML、表格等格式：

请生成3本虚构书籍，以JSON格式输出，包含以下字段：
book_id, title, author, genre

策略3：条件判断

让模型先检查条件再执行：

如果文本中包含操作步骤，请用编号列出。
如果文本中没有操作步骤，请回复"未提供步骤"。

策略4：少量示例（Few-Shot）

给1-2个示例让模型模仿风格：

用一致的风格回答：
孩子：教我什么是耐心。
祖父：让河流雕刻出山谷的那种力量...
孩子：教我什么是韧性。

💡 关键认知：分隔符是最简单但最有效的改进。很多"模型不听话"的问题，加个分隔符就解决了——因为模型终于能分清"哪是指令、哪是内容"。

原则二：给模型足够的思考时间

一句话：复杂问题不要让模型"脱口而出"，而是让它一步步推理。

📐 给模型思考时间的两大策略

两大策略

策略1：指定完成步骤

请按以下步骤执行：
第1步：用一句话概括以下文本。
第2步：将概括翻译成法语。
第3步：列出法语概括中的人名。
第4步：输出包含"法语摘要"和"名字数量"的JSON。

策略2：让模型先自己解题，再评判

课上的经典例子——太阳能发电站维护成本： - 学生把"10美元/平方英尺"的维护成本写成了"100x"（应该是"10x"） - 如果直接问模型"学生做对了吗"，模型可能匆忙说"对的" - 但如果先让模型自己解题，再对比学生的答案——模型就能准确发现错误

💡 关键认知：这就是"思维链"（Chain of Thought）的雏形。让模型展示推理过程，准确率会大幅提升。人类考试也是一样——先打草稿再答题，比直接写答案靠谱得多。

二、迭代式Prompt开发：方法论

一句话：没有人能一次写出完美的Prompt。核心方法是：写→测→分析→改→再测，循环迭代。

📐 迭代式Prompt开发循环

椅子产品描述的迭代案例

这是课上最完整的迭代示例：

初始Prompt："根据产品说明书写一段营销描述"

问题1：生成文本太长（500字） → 改进：加约束"最多50个词"

问题2：关注了错误的细节（说了太多尺寸信息） → 改进：指定"侧重材料和技术细节，面向家具零售商"

问题3：需要表格形式 → 改进：要求"用HTML表格输出，包含产品名、材料、尺寸"

迭代循环：
写初版Prompt → 测试输出 → 分析问题（太长？角度错？格式不对？）
     ↑                              ↓
     └──── 修改Prompt（应用两大原则） ←┘

💡 关键认知：Prompt不是"碰运气"，而是可以工程化的。就像软件开发一样——写代码、跑测试、找Bug、改代码——Prompt开发也是同样的迭代循环。

三、四大应用实战

📐 四大应用能力递进

应用一：文本摘要（Summarizing）

关键技巧：用"提取"比"总结"更精准。

从以下评论中提取与运输和发货相关的信息，限30字以内。

课上例子：电商评论 → 分别为物流部门、定价部门、营销部门定制不同视角的摘要。

摘要技巧做法效果
限定字数"用30个词总结"控制输出长度
限定视角"从物流部门的角度总结"关注特定维度
提取vs摘要"提取关于价格的信息"精准信息抽取

💡 关键认知：摘要和提取(extract)是两回事。摘要是"用更短的话复述"，提取是"只拿出特定信息"。很多时候你以为需要摘要，实际上需要的是提取。

应用二：文本推理（Inferring）

情感分析：

判断以下评论的情感倾向，只回答"正面"或"负面"：

信息提取：

从评论中提取购买的商品和制造商，以JSON格式输出，
键为"Item"和"Brand"。

课上例子：台灯评论 → 判断正负面 → 识别具体情绪（满意、愤怒等，最多5种）→ 判断是否包含愤怒。

推理任务Prompt示例输出类型
情感分析"这条评论的情感是正面还是负面？"正面/负面/中性
主题识别"这段文字涉及哪些主题？从列表中选"标签列表
实体提取"提取文中的品牌名和价格"结构化JSON
多任务并行"同时做情感分析+主题识别+实体提取"一次性输出多维结果

应用三：文本转换（Transforming）

  • 翻译：翻译成法语、西班牙语、甚至"英语海盗话"
  • 语气调整：俚语 → 正式商务信函
  • 格式转换：Python字典 → HTML表格
  • 校对改写：检查拼写语法，按APA风格改写
转换任务输入输出关键技巧
翻译英文邮件中文邮件指定"正式/口语"语气
语法修正有语法错误的文本修正后的文本"用markdown标注修改处"
格式转换JSON数据HTML表格给出目标格式的示例
语气调整随意的消息正式商务邮件指定目标读者

应用四：文本扩展（Expanding）

核心能力：根据少量输入生成长文本。

课上例子：根据客户评论自动生成回复邮件 - 正面评论 → 致谢 + 邀请复购 - 负面评论 → 道歉 + 建议联系客服

Temperature参数在这里登场： - T=0：每次生成同样的回复（确定性） - T=0.7~1.0：每次生成不同风格的回复（创意性）

💡 关键认知：四大应用的递进关系——摘要是"压缩信息"，推理是"深层理解"，转换是"改变形式"，扩展是"创造内容"。从理解到创造，难度逐步升级。

💡 关键认知：扩展(Expanding)是最需要注意Temperature的场景。Temperature=0时输出千篇一律（适合摘要），但扩展需要适度创意，建议Temperature=0.7左右。课上演示了用低温和高温写同一封客服邮件，效果天差地别。

四、聊天机器人：从单次Prompt到多轮对话

一句话：模型本身是无状态的——它不"记得"之前的对话。所谓的"记忆"是每次把完整对话历史重新发一遍。

📐 聊天机器人: system/user/assistant三角色对话流

三种角色

角色 作用 谁设定的
system AI的"人设"和行为准则 开发者（用户看不到）
user 用户的消息 用户
assistant 模型的回复 模型生成

关键实现

每次API调用都要传完整的对话列表：

messages = [
    {"role": "system", "content": "你是友好的AI助手"},
    {"role": "user", "content": "你好"},
    {"role": "assistant", "content": "你好呀！"},
    {"role": "user", "content": "我叫什么？"}  # 需要靠历史记录
]

💡 关键认知：模型不是真的"记住"了你——它只是每次都重新阅读完整的聊天记录。这就是为什么对话越长、Token消耗越多、成本越高的原因。也解释了为什么长对话后模型会"忘事"——因为历史太长被截断了。

五、课程的隐含最佳实践

  1. 始终用分隔符——最简单有效的改进
  2. 明确输出格式——让模型输出JSON/表格便于程序处理
  3. 提供1-2个示例——比长篇说明更有效
  4. 拆解复杂任务——一个复杂Prompt拆成多步
  5. 建立Prompt模板库——按场景分类存储，团队复用
  6. 用A/B测试验证——不凭感觉，用数据说话

六、核心洞察

📐 幻觉问题: LLM为什么会"一本正经地胡说"

幻觉问题

模型可能"一本正经地胡说八道"。对策：让模型先从文本中收集信息，再基于信息回答——而不是凭"记忆"生成。

"提取"比"总结"好

使用关键词"提取"（extract）让模型聚焦特定信息范围，比笼统的"总结"（summarize）效果更好。

从尝试走向工程化

Prompt不是"试试运气"，而是可复用、可标准化的工程产物。应建立模板库，按场景分类，用测试数据量化效果。
