内容综述

这期视频讲的是一个比“写好提示词”更底层的问题：模型到底应该看见什么。如果把大模型想成一个函数，你给它输入，它给你输出；但这个输入不是无限大的，能放进去的所有指令、资料、历史对话、工具结果，都要挤在一个叫 Context Window（上下文窗口） 的地方。

Context Engineering（上下文工程）解决的不是“怎么把 Prompt 写得更漂亮”，而是“怎么把下一步真正需要的信息放进窗口里”。这件事对 Agent 特别重要，因为 Agent 会反复思考、调用工具、读文件、搜索网页、执行代码，工具结果越滚越多，最后很容易出现上下文污染、干扰、混乱和冲突。

视频的主体可以归纳成四个动作：保存 Context、选择 Context、压缩 Context、隔离 Context。保存是把重要信息放到窗口外；选择是每一步只拿相关材料；压缩是把冗长材料提炼成高信号摘要；隔离是用多 Agent、沙箱或状态字段，把不同任务的上下文分开管理。

第一部分：Context 不是“聊天记录”，而是模型的工作台

📐 图1：Context 是模型当前工作台，不只是聊天记录

一、Context：模型这一次能看到的全部东西

一句话：Context 就是模型生成答案时“摆在桌面上的全部材料”。

很多人会把 Context 理解成“上文”或“聊天记录”，这个理解太窄了。对一个 AI 应用来说，Context 至少包括：

  • 系统指令：你是谁、要遵守什么规则、输出什么风格；
  • 用户输入：用户这一次真正问了什么；
  • 历史对话：前面聊过什么，有哪些承诺和约束；
  • 外部知识：文档、网页、数据库、FAQ、代码片段；
  • 工具列表：模型现在能调用哪些工具；
  • 工具结果：搜索结果、API 返回、代码执行输出、文件内容；
  • 记忆信息：用户偏好、项目约定、历史经验；
  • 输出格式要求：比如必须输出 JSON、表格、报告或代码补丁。

换句话说，Context 不是一句 Prompt，而是一个“临时工作台”。模型只能基于工作台上已经摆出来的东西做判断；没有摆出来的信息，它通常不会凭空可靠地知道。

用户问题 + 系统指令 + 历史对话 + 外部资料 + 工具结果 + 记忆
                         ↓
                  Context Window
                         ↓
                       LLM
                         ↓
                    当前这一步输出

💡 关键认知：Prompt 是你对模型说的话，Context 是模型这一步能看见的整个世界。只优化 Prompt，不管理 Context，就像只改会议主持词，却不准备会议材料。

二、Context Window：不是无限仓库，而是有限内存

📐 图2：大上下文窗口不等于可以无脑堆料

一句话：Context Window 是模型一次调用最多能处理多少 token 的上限。

视频里用一个很直观的说法解释 token：它是文本被拆出来的最小单位，可能是一个字、一个词，也可能是一个标点。粗略理解，1 个 token 约等于 0.75 个英文单词，或 1.5 个汉字。

比如 Gemini 2.5 Pro 的上下文窗口可以达到 100 万 token，听上去很大，接近几本书的长度。于是很多人会自然产生一个想法：既然窗口这么大，那是不是把所有资料都塞进去，让模型自己找重点就行？

这正是 Context Engineering 要反对的“偷懒做法”。窗口大，只代表“能塞”；不代表“塞进去就会用得好”。

做法 表面好处 实际问题
把所有资料都塞进去 简单，不用做检索和筛选 成本高、延迟高、模型分心、重点被稀释
只塞用户问题 便宜、快 缺少背景，容易泛泛而谈或幻觉
按任务动态构造 Context 信息密度高 需要工程设计：检索、排序、压缩、隔离

💡 关键认知：大上下文窗口像大桌子，不是垃圾场。桌子越大，越需要会摆东西；否则东西越多，越找不到重点。

第二部分：从 Prompt Engineering 到 Context Engineering

📐 图3：Prompt Engineering 与 Context Engineering 的分工

一、Prompt Engineering 解决“怎么问”，Context Engineering 解决“让模型看什么”

一句话：Prompt Engineering 是话术设计，Context Engineering 是信息环境设计。

Prompt Engineering 的关注点通常是：

  • 角色怎么设定；
  • 指令怎么写；
  • 示例怎么给；
  • 输出格式怎么约束；
  • 语气和步骤怎么提示。

这些当然有用，但它解决不了一个根本问题：如果模型没有正确材料，再漂亮的 Prompt 也只能让它更自信地猜。

Context Engineering 的关注点更靠近工程系统：

  • 哪些文档要被检索出来；
  • 哪些历史对话应该保留；
  • 哪些工具结果应该注入；
  • 哪些旧信息应该压缩或丢弃；
  • 哪些上下文应该只给子 Agent 看，不给主 Agent 看；
  • 哪些信息不能混在一起，避免冲突或泄露。
维度 Prompt Engineering Context Engineering
核心问题 怎么把指令写清楚 怎么让模型拿到正确材料
主要对象 Prompt 文本 指令、知识、工具结果、记忆、状态
典型手段 角色设定、Few-shot、格式约束 RAG、Memory、压缩、多 Agent、状态管理
失败原因 指令含糊 材料缺失、太多、错误、冲突
更像什么 写一份好题目 给考生准备正确资料包

💡 关键认知：Prompt Engineering 是“问得好”，Context Engineering 是“喂得准”。Agent 真正落地时，后者往往决定 80% 的效果。

二、Karpathy 的 CPU/RAM 类比

LangChain 文章引用 Andrej Karpathy 的类比：

Context engineering is the delicate art and science of filling the context window with just the right information for the next step.

中文可以翻成：上下文工程是一门精细的艺术与科学，它要在 Agent 的每一步，把“刚刚好”的信息放进上下文窗口。

这个定义里最重要的词不是 information，而是 just the right information。不是更多，也不是更少，而是“下一步刚好需要”。

可以这样理解：

传统程序：CPU 处理 RAM 里的数据
AI Agent：LLM 处理 Context Window 里的信息

操作系统要管理内存：加载、缓存、淘汰、隔离
Agent 系统要管理 Context：保存、选择、压缩、隔离

如果把 LLM 看成 CPU，把 Context Window 看成 RAM，那么 Context Engineering 就像“内存管理”。没有内存管理，复杂程序跑不稳；没有上下文管理，复杂 Agent 也跑不稳。

第三部分：为什么长上下文会让 Agent 变差

📐 图4：长上下文的四类失败模式

Agent 和普通聊天最大的区别是：它不是问一次答一次，而是会持续执行任务。

典型 Agent 轨迹如下：

思考 → 调用搜索 → 读取网页 → 总结 → 调用数据库 → 生成草稿 → 执行代码 → 修复错误 → 输出结果

每一步都会产生新 Context。工具返回很长，网页内容很长，代码报错很长，历史决策也越来越多。如果不管理，Context 会像滚雪球一样变大。

一、Context Poisoning：上下文污染

一句话：错误信息一旦进入 Context，后面每一步都可能被它带偏。

比如 Agent 第一次搜索时误读了一个字段，把“订单已取消”总结成“订单已支付”。如果这个错误总结被写进 Memory，后续所有回答都会继续引用这个假事实。

污染最麻烦的地方是：它会伪装成“已有上下文”。模型看到它时，并不会自动知道这是错的。

💡 关键认知：上下文污染比一次回答错误更危险，因为它会变成后续推理的地基。地基错了，楼越盖越歪。

二、Context Distraction：上下文干扰

一句话：材料太多时，模型会被无关信息分散注意力。

这就像你让一个新人客服回答“这个订单为什么没发货”，却把公司全部产品手册、所有历史邮件、全量客户聊天记录都给他。信息是多了，但任务变难了。

模型不是数据库查询引擎，它不会天然总能在海量噪声里稳定抓住重点。长上下文会增加成本和延迟，也会降低答案稳定性。

三、Context Confusion：上下文混乱

一句话：多余信息不一定是错的，但会让模型用错材料。

比如用户问“帮我按上次销售周报的格式写一版”，Context 里却同时塞入了技术方案、招聘 JD、客户投诉模板。模型可能会混合风格，最后生成一个四不像。

四、Context Clash：上下文冲突

一句话：不同上下文互相打架时，模型不知道该信谁。

常见冲突包括：

  • 系统指令要求短回答，用户要求详细报告；
  • 旧记忆说用户喜欢 A，新消息说这次要 B；
  • 一个文档说价格是 99，另一个文档说价格是 129；
  • 工具 A 返回成功，工具 B 返回失败。
失败模式 核心问题 典型后果 应对方向
Context Poisoning 错误进入上下文 后续推理持续错误 写入前验证、来源标注
Context Distraction 信息太多 忽略重点、成本升高 选择、排序、裁剪
Context Confusion 多余信息影响回答 风格/判断跑偏 精简 Context、分任务加载
Context Clash 信息互相矛盾 输出不稳定 优先级、版本、权威源

第四部分：Context Engineering 的四个基本动作

📐 图5：四个动作构成上下文工程闭环

一、保存 Context：把重要信息放到窗口外

一句话：保存不是把所有东西一直塞给模型，而是把有价值的信息放在外部，需要时再取。

视频把第一类实现方法概括为“保存 Context”。LangChain 文章里对应的是 Write Context。

常见保存对象有两类：

  1. Scratchpad（草稿纸）：当前任务内的临时记录，比如计划、待办、关键观察、中间结论；
  2. Memory（记忆）：跨任务、跨会话的长期信息，比如用户偏好、项目约定、成功/失败经验。
不保存：
所有计划、工具结果、观察都堆在 Context Window 里
→ 超长、易丢、易乱

保存后：
关键状态写到外部 state / file / database / memory
→ 当前窗口只放下一步需要的内容

保存 Context 的关键不是“记得越多越好”，而是要有写入标准：

  • 这条信息未来还会用到吗？
  • 它是否经过验证？
  • 它的来源和更新时间是什么？
  • 它应该对所有任务可见，还是只对当前任务可见？

💡 关键认知：Memory 不是收藏夹，而是会影响模型判断的“事实库”。写进去之前要谨慎，否则就是主动制造上下文污染。

二、选择 Context：每一步只拿相关材料

📐 图6：RAG 是 Context Selection 的典型实现

一句话：选择 Context 就是从外部世界取回“当前这一步最相关”的信息。

这一步最常见的实现是 RAG（Retrieval-Augmented Generation，检索增强生成）。用户问问题时，系统先去知识库里找相关文档片段，再把这些片段塞进 Context Window，让模型基于材料回答。

但选择 Context 不只等于 RAG，还包括：

  • 选择哪些 Memory 被加载；
  • 选择哪些工具描述暴露给模型；
  • 选择哪些历史对话保留；
  • 选择哪些项目规则、代码文件、数据库记录进入当前步骤；
  • 选择静态 Context 和动态 Context 的比例。
选择对象 静态选择 动态选择
指令 系统提示词、角色规则 根据任务切换不同规则
知识 固定业务说明、产品手册 RAG 检索相关文档
工具 默认工具列表 根据任务检索相关工具
记忆 固定用户偏好 根据问题召回相关记忆
历史 最近几轮对话 按主题/实体筛选历史

💡 关键认知：RAG 的本质不是“搜索一下”，而是 Context Selection：在模型回答前，决定它应该看哪几块材料。

三、压缩 Context：用更少 token 保留更高信号

📐 图7：压缩 Context：从低密度原文到高密度状态

一句话：压缩不是简单变短，而是把低密度材料变成高密度材料。

当工具结果、聊天历史、网页资料太长时，直接塞给模型会浪费窗口。压缩 Context 有三类常见做法：

  1. 摘要压缩：把长对话、长文档、长工具输出总结成短摘要；
  2. 结构化抽象：把自然语言变成字段、标签、任务状态、决策记录；
  3. 修剪裁剪：删除过早、无关、重复、格式噪声很高的信息。

比如一个客服 Agent 不需要每次都看到用户过去 100 条聊天原文，它可能只需要：

用户画像：企业客户，偏好邮件沟通
当前问题：订单 8472 未发货
历史约束：上次承诺 2 个工作日内反馈
待办事项：查询物流系统，确认是否缺货
压缩方式 适合场景 风险
摘要 Summarization 长对话、长搜索结果 细节丢失、总结偏差
修剪 Trimming 旧消息、重复内容 删掉关键约束
结构化抽象 任务状态、客户画像 字段设计不完整
专用压缩模型 高频企业流程 成本和维护复杂度更高

💡 关键认知：压缩 Context 的目标不是“少”，而是“信号密度更高”。该保留的约束、数字、例外和来源，不能被摘要吃掉。

四、隔离 Context：别让所有任务共用一个脑袋

📐 图8：隔离 Context：多 Agent、沙箱与状态字段

一句话：隔离 Context 就是把不同任务、不同角色、不同状态拆开，避免互相污染。

隔离最典型的方式是 Multi-Agent。比如做深度研究时，一个子 Agent 负责找论文，一个负责找产品案例，一个负责找代码实现，最后主 Agent 只接收它们压缩后的结论。

这样做的好处是：

  • 每个子 Agent 的上下文窗口更干净；
  • 不同主题不会互相干扰；
  • 可以并行探索；
  • 主 Agent 不需要阅读所有原始材料。

隔离也可以发生在执行环境里。比如代码执行、图片处理、音频处理、大型表格分析，都可以放在沙箱或程序状态中，最后只把结果摘要返回给 LLM。

主 Agent：只看任务目标、关键结论、最终冲突
子 Agent A：只看论文资料
子 Agent B：只看产品案例
子 Agent C：只看代码仓库
沙箱环境：保存大文件、中间变量、执行状态

💡 关键认知：不是所有信息都应该给同一个模型看。隔离上下文，本质是在给 Agent 系统划分“办公室”和“资料权限”。

第五部分：把四个动作放进一个真实系统

一、以智能客服为例：Context 应该怎么流动

视频里提到一个很容易理解的场景：如果要做一个智能客服，希望它能回答用户关于公司产品的问题，不能只给它一句“你是客服，请回答用户问题”。

一个更像样的 Context Engineering 流程应该是：

用户问题
  ↓
识别意图与实体：产品名、订单号、问题类型
  ↓
选择 Context：检索产品文档、FAQ、订单状态、用户历史
  ↓
压缩 Context：只保留相关条款、时间、异常原因、处理规则
  ↓
隔离 Context：订单系统结果、知识库结果、投诉历史分开处理
  ↓
生成回答：引用依据，给出下一步动作
  ↓
保存 Context：记录本次承诺、用户偏好、未完成事项

如果没有这套流程，客服 Agent 很容易出现三种问题：

  • 用户问具体订单，它回答通用政策；
  • 用户问新版产品，它引用旧版文档；
  • 用户之前已经投诉过，它完全不知道历史背景。

二、工程实现时可以用一张检查表

做 AI Agent 时，可以把每次 LLM 调用前的 Context 当成一个“资料包”，用下面这张表检查。

检查项 要问的问题 对应动作
目标清楚吗 这一步到底要模型做什么？ Prompt / 指令
材料相关吗 当前资料是否直接服务这一步？ Select
材料可信吗 来源、时间、版本是否可靠？ Write 前验证
材料太长吗 有没有重复、噪声、低价值内容？ Compress
材料冲突吗 有没有新旧版本、口径不一致？ 优先级 / 规则
边界清楚吗 哪些信息不该混进来？ Isolate
结果要记吗 本次输出是否影响未来任务？ Write

💡 关键认知：每次调用 LLM 前，都应该问一句：“这一步的 Context 是不是刚好够用？”不是“能不能再塞一点”。

三、一个最小可落地架构

如果只做一个实用版本，不需要一上来就多 Agent。最小可落地架构可以是：

  1. State：保存任务目标、当前阶段、关键约束、待办；
  2. Retriever：从知识库、文档、历史记录里选择相关材料；
  3. Compressor：对长材料做摘要、字段化、去重；
  4. Prompt Builder：把指令、用户问题、相关材料组装成 Context；
  5. Verifier：检查输出是否引用了材料、是否遗漏约束；
  6. Memory Writer：把经过确认的偏好、承诺、结论写回记忆。
External Sources ──→ Retriever ──→ Compressor ──→ Prompt Builder ──→ LLM
       ↑                                                        ↓
       └────────────── Memory Writer ← Verifier ← Output ──────┘

这个架构不花哨，但抓住了 Context Engineering 的主线：外部保存、按需选择、进入前压缩、必要时写回。

第六部分：学习这期视频后，应该记住什么

Context Engineering 的核心不是一个新名词，而是一种工程视角：LLM 的能力，要通过正确的信息环境才能发挥出来。

如果说 Prompt Engineering 是“把需求说清楚”，那么 Context Engineering 是“把证据、状态、工具和边界准备好”。对简单聊天来说，Prompt 可能够用；对 Agent、RAG、企业知识库、智能客服、代码助手来说，Context 才是主战场。

最后可以用四句话收束：

  1. 保存：重要信息不要全堆在窗口里，写到外部状态和记忆；
  2. 选择：每一步只拿相关材料，RAG 是选择的一种实现；
  3. 压缩：长材料先提炼，提高信号密度，保留关键约束；
  4. 隔离：不同任务、角色、工具结果分开处理，避免互相污染。

💡 关键认知：上下文工程的目标不是让模型“知道更多”，而是让模型在每一步“刚好知道该知道的”。
