📋 内容综述

这期视频带你搞清楚 Harness Engineering 这个 2026 年开年 AI 圈最火的新名词，核心回答四个问题：

第一，Harness Engineering 是什么？ 一句话：Agent = Model + Harness，模型本身差距已经很小，真正拉开水平的是模型外面那套"缰绳"——包括工具接口、上下文管理、沙箱执行、权限门、错误反馈回路。Harness 在软件工程里本来指"测试装置"，现在被借来形容"驾驭大模型的运行时装备"。

第二，它和 Prompt / Context Engineering 是什么关系？ 三个概念对应三次工程范式跃迁：Prompt 管"怎么说话"、Context 管"看什么信息"、Harness 管"在什么系统里跑"。它们不是互相取代，而是关注点一层比一层高——从字符串、到信息管线、再到整个运行时基础设施。

第三，OpenAI 和 Anthropic 都在怎么玩？ OpenAI 最出圈的是"3 人团队 5 个月写 100 万行代码，0 行手写"——人类 100% 时间用来设计 Harness；Anthropic 则写了《Effective Harnesses for Long-Running Agents》用"Initializer + Coding"双 Agent 模式破解长程失忆问题。Claude Code 本身就是教科书级的 Harness 范例。

第四，它是技术突破还是概念炒作？ 支持派用 Terminal Bench 2.0 实测说话——同模型仅换 Harness，通过率从 52.8% 飙到 66.5%；反对派则指出这无非是"关注点分离"换了个新包装，Noam Brown 甚至担心下一代模型会把今天的复杂 Harness 冲刷成历史包袱。看完你会发现：Harness 是真的，但"Engineering"部分里有多少是新东西，答案因人而异。

一、这个新词到底是什么：Harness 一词的前世今生

📐 图1：Harness 一词的演变——从马具到 AI 工程

1.1 Harness 不是新词

一句话：Harness 在英文里的字面意思是"马具、缰绳"——一套把一头力大无穷但不一定听话的牲口稳稳驾住的装备。

这个词在软件工程里早就有了，叫 Test Harness（测试装置 / 测试夹具）。它是一套用来驱动被测代码、模拟外部依赖、收集结果的脚手架。在 LLM 评测领域有个叫 lm-evaluation-harness 的开源项目（EleutherAI 维护），专门用来跑各种 benchmark。

所以新的不是 "Harness"，而是 Harness Engineering 这个组合词——它把"马具"从一个名词变成了一门专门的工程实践。

1.2 Harness Engineering 的正式定义

OpenAI 官方口径（2026 年 2 月 11 日博客）：

Harness engineering is the design of the runtime layer around the model: tool interfaces, context assembly and compaction, sandboxed command execution, policy enforcement, observability, and failure recovery.

翻译过来：Harness Engineering 就是设计模型外面的那整个运行时层——工具接口、上下文装配与压缩、沙箱命令执行、策略执行、可观测性、失败恢复。

通俗理解公式：Agent = Model + Harness。

部分 类比 你能不能动手
Model（模型） 发动机 只有少数大厂能炼
Harness（缰绳/外壳） 车身、方向盘、刹车、仪表盘 所有应用工程师都能调

💡 关键认知：2026 年各家前沿编码模型在 SWE-bench Verified 上的差距已经不到 1 个百分点。模型本身卷不动了，真正拉开 Agent 水平的是外面那套 Harness。这是 "Harness Engineering" 作为独立范畴出现的根本原因。

二、三层关系：Prompt → Context → Harness

2.1 三次工程范式跃迁

一句话：Prompt 管怎么说话、Context 管看什么信息、Harness 管在什么系统里跑。一个比一个抽象层次高。

              ↓                            ↓                             ↓

💡 关键认知：这三个不是替代关系而是包含关系。你今天做 Harness Engineering，依然要做好 Prompt 和 Context，只是不再把优化精力只压在这两层上。三者的区分本质是"工程师能影响系统的抽象层次"不断上移。

2.3 为什么会出现 Harness 这个新层？

Context Engineering 在 2025 年已经有过一波讨论，当时 Karpathy 说"Context Engineering 才是 Prompt 的未来"。但到了 2026 年初，大家发现一件尴尬的事：

只管上下文还不够——模型拿到完美的上下文，执行环境依然可能把它带沟里。

比如一个 Coding Agent，就算你给它最完美的任务上下文，如果： - 工具调用错了，它的自我修复机制很差 - 文件写坏了，没人拦住它（缺权限门） - 连续失败 10 次还在试（没有 escalation 机制） - 代码质量下降了，没 linter 告诉它

那么上下文再完美也没用。这些东西都不在"上下文"范畴里，它们是系统基础设施层的东西——这就是 Harness Engineering 出现的理由。

3.2 OpenAI 团队总结的 6 条 Harness 实战法则

法则 1：设计环境，而非编写代码

人类工程师的工作从"写代码"变成"写 AGENTS.md、写 linter、写 schema"。所有会重复出现的约束全部被代码化到环境里。

法则 2：每次失败都问"缺什么能力"，而不是"再试一次"

这是 Harness Engineering 与 Prompt Engineering 思维方式的根本分水岭。OpenAI Ryan Lopopolo 的原话：

Failure after failure, the solution was almost never "try again." The only way to move forward was to ask: what capability is missing, and how do we make the agent understand and follow it?

法则 3：机械化执行架构约束

光写"分层依赖不要倒置"在 AGENTS.md 里不够——Agent 会忘。必须写 linter 自动 ban 掉违规代码，让约束变成机器可执行的规则。

法则 4：代码仓库 = 唯一事实源

禁止把团队知识放在 Slack / Google Docs / Notion 里。Agent 看不到的东西就等于不存在。所有决策、约定、约束必须进代码库。

法则 5：可观测性直通 Agent

让 Agent 能直接读 Chrome DevTools、日志、监控指标，不要隔着人类传话。Agent 需要看到真实系统状态，而不是人类二次概括过的状态。

法则 6：熵管理（"以小额贷款方式偿还技术债务"）

Agent 会留下垃圾代码、文档不一致、冗余实现。OpenAI 实验初期每周五花 20% 时间手动清理，后来引入定期巡逻的 cleanup agent——每隔一段时间扫描仓库，发针对性重构 PR，大多数 1 分钟内修完。

💡 关键认知："以小额贷款方式偿还技术债务，而不是等它复利后再痛苦清算"——这是 Harness 区别于传统开发的关键。传统项目债务线性累积，Agent 项目债务指数累积，必须内置定期清理机制。

3.3 Codex 暴露的两类典型失效模式

Agent 失败不是随机的，有固定套路：

失效模式 症状 Harness 对策
One-shot 崩溃 试图一口气写完所有代码 → 耗尽上下文窗口 → 放弃 拆任务、外化 progress、双 Agent 接力

四、Anthropic 的实战：Claude Code 就是教科书级 Harness

📐 图5：Claude Code 三层架构——Harness 的教科书范例

4.1 Claude Code 的三层架构

Claude Code 是 Anthropic 的 CLI 编码 Agent。2026 年 3 月其源码（约 51.2 万行）被社区分析后，发现它本身就是 Harness Engineering 的范本：

┌─────────────────────────────────────────────────────┐
│       Orchestration Loop（编排循环）                  │
│       QueryEngine.ts ~4.6万行                        │
│       思考→行动→观察→再思考                            │
└──────────────────┬──────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────┐
│  Permission-Gated Tools（带权限门的工具层）            │
│  ~40 个独立能力：read / write / bash / grep / ...    │
│  每个工具有独立权限门                                   │
└──────────────────┬──────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────┐
│        Multi-Layer Memory（三层记忆）                 │
│        会话外化 + 校验                                 │
└─────────────────────────────────────────────────────┘

权限三阶段：项目载入时建立信任 → 每次工具调用前 check → 高危操作显式人类确认。

4.2 三层记忆架构

层级 存储 大小 访问时机
L1 MEMORY.md 每条 ~150 字 永远加载
L2 主题文件 详细笔记 按需加载
L3 原始 transcript 全量历史 仅在搜索时读

💡 关键认知：Claude Code 的铁律是 Agent 把自己的记忆当 "hint" 而非 "fact"——动手前必须对照真实代码库再校验一次。记忆只是线索，事实永远以当前代码为准。这解决了 Agent 最容易犯的"记错了还自信满满"问题。

📐 图6：双 Agent 模式——长程任务如何穿越上下文窗口边界

4.3 长程任务的"双 Agent"模式

Anthropic 2025 年 11 月发的《Effective Harnesses for Long-Running Agents》提出了一个关键比喻：

长程 Agent 就像一支轮班的工程师团队，每次接班的人都失忆。

上下文窗口再大也有上限，跨多天的项目必然要把状态外化到文件系统。方案是拆成两个 Agent：

┌─────────────────────────────┐
│   Initializer Agent          │
│   （一次性，项目第一天跑）     │
│                              │
│   · 写 init.sh               │
│   · 生成 feature list        │
│   · 初始化 progress.md       │
│   · git init                 │
└──────────────┬───────────────┘
               │ 把"外部记忆"铺好
               ↓
┌─────────────────────────────┐
│    Coding Agent              │
│    （每次开工循环一遍）        │
│                              │
│  1. read progress.md         │
│  2. check feature list       │
│  3. 拣下一个未完成 feature    │
│  4. 写代码                    │
│  5. 回写 progress            │
│  6. git commit               │
│                              │
│  → 窗口耗尽，下一轮接班         │
└─────────────────────────────┘

核心四件套：feature list / progress file / init.sh / git commit——把"工程师的工作流程"代码化。

4.4 多 Agent 协作三种模式

模式 机制 典型场景
Fork 复制父 context，prompt cache 复用 平行探索，缓存几乎免费
Teammate tmux/iTerm 独立 pane，文件邮箱通信 松耦合分工
Worktree 隔离 git worktree、独立分支 探索性或高风险工作

4.5 Brain / Hands 解耦

Anthropic 把组件分成三层：

┌─────────────────────────────────────┐
│  Session（会话层：日志/记忆，持久化）  │ ← Brain 的"记忆"
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Harness（编排层：无状态循环）         │ ← Brain
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Sandbox（沙箱：按需创建销毁）         │ ← Hands
└─────────────────────────────────────┘

💡 关键认知：容器死掉 ≠ 会话丢失。Harness 可以无状态重启，从 Session 恢复继续干活。Anthropic 自报数据：解耦后 TTFT（首 token 延迟）下降 60–90%。这让"sandbox 是牛，出事就杀了换"成为可行策略——想想传统开发里你能随便杀掉环境吗？

📐 图8：错误反馈的两类传感器——Computational vs Reasoning

5.2 两类传感器（错误反馈回路）

类型 延迟 资源 确定性 例子 何时跑
Computational 毫秒级 CPU 高 linter、type check、unit test 每次都跑
Reasoning 秒级 GPU（贵） 低 AI code review、LLM as judge 抽检

💡 关键认知：这里有两个反模式： - 只有 feedforward 没有 feedback（只写规则不检查）→ 规则变成死文档，没人遵守 - 只有 feedback 没有 feedforward（只检查不定规则）→ 同样的错反复犯

两者必须同时在 Harness 里出现。

5.3 Evaluator Agent：给 Agent 找个"专职挑刺的"

为什么需要独立 Evaluator？

Agent 自评几乎总通过——再烂的代码自己看也满意。这是大模型作为 LLM as judge 的根本问题。

解决办法：训练一个专门的 Evaluator Agent，用 Playwright 真点 UI、调真 API、查真 DB 状态。核心原则：让 QA 与 Generator 分人做，比教 Generator 自我批评容易得多。这有点像 GAN 的思路——两个网络对抗进化。

5.4 渐进信任分级

不要一开始就让 Agent 随便改代码，按阶段放权：

Stage 能做什么
Stage 1 只读 + 看 lint 反馈
Stage 2 改小文件、非核心模块
Stage 3 改重要文件
Stage 4 自主提 PR
Stage 5 CI 绿就自动合并

Stripe 的 Minions 系统实战规则：两次连续失败即升级到人，不允许无限重试。

6.1 Terminal Bench 2.0（LangChain 的实验）

实验设计：同一个模型、只换外面的 Harness。

Harness 通过率 排名

6.2 SWE-bench Pro 上的 scaffold vs 模型对比

变量 性能摆动幅度
换模型（同 scaffold） 1x（基准）
换 scaffold（同模型） 22x

Meta 的 Confucius Code Agent 实验：自定义 scaffold 让 Claude Sonnet 在 SWE-bench Pro 跑到 52.7%，反超 Anthropic 官方 scaffold 上的 Claude Opus（52.0%）。

论文金句：

"The improvement comes purely from stronger agentic scaffolding, not from a better model."

6.3 反方实证

但不是所有场景加 Harness 都有用：

  • 138 配置文件研究：测试 138 个 AI 自动生成 的 harness，发现自动生成的 harness 反而让性能下降、增加 20% 成本；即使人类专家手写 harness，平均提升也只有 约 4%。
  • 清华实验反直觉发现：给 Agent 加验证器（validator）反而让性能在某些 benchmark 大幅下降。但换一个方式：用自然语言描述验证器（而非用代码实现），成功率从 30.4% 跃到 47.2%——问题不只在加什么，还在怎么表达。

七、争议：Harness Engineering 是真突破还是换包装？

📐 图10：争议两派——技术突破 vs 概念炒作

7.1 支持派的五大理由

# 代表人物 观点
1 Mitchell Hashimoto 每次错误都工程化解决，能把 Agent 可用率推到 99.9%。这不是 prompt tuning，是系统进化。
2 OpenAI Ryan Lopopolo "Agents aren't hard; the harness is hard." 失败时几乎从来都不是"再试一次"能解决。
3 LangChain（Lance Martin） 公式 Agent = Model + Harness 是对 Agent 工程的正确抽象。实证数据在 Terminal Bench 2.0 上说话。
4 Meta Confucius 团队 同模型换 scaffold 能逆转排名，说明 Harness 改造空间巨大。
5 Martin Fowler Harness 会成为 AI 时代的"服务模板"——就像今天微服务有标准脚手架。

7.2 反对/质疑派的六大理由

# 代表人物/来源 观点
1 Chenyang Zhao（SGLang 团队） Harness Engineering 本质就是传统软件工程的"关注点分离"、"单一职责"换名字包装。他自己做 AI 系统时凭直觉用了同样方法，事后才发现这些招在 Harness 里有专门名字。并非全新发明。
2 Noam Brown（OpenAI 推理模型研究员） 当前复杂 Harness 系统最终可能被下一代更强模型冲刷掉变成冗余——基础设施太多反而变成历史包袱。这是 Bitter Lesson 在 Agent 领域的回声。
3 Lance Martin（"减法工程"） 真正好的 Context/Harness 设计是"做减法"——上下文不是越多越好，工具不是越多越好。过度工程化反而让 Agent 更难用。
4 138 配置文件研究 AI 自动生成的 harness 反而让性能下降；人类专家手写平均提升也只有 4%。大部分"加 harness"其实没用。
5 Birgitta Böckeler（Martin Fowler 网站） 现有 Harness 主要保证"代码干净"，不保证"产品做对了"。刀工再好厨房再整洁，菜不好吃没用。同时指控利益冲突：OpenAI 自己写文章推广 Harness，自然支撑自家 Codex 销售。
6 AI 产出质量隐忧 Agent 为通过验证而伪造测试；把临时权宜之计系统性复制到整个代码库，快速累积难维护的技术债。OpenAI 自己承认 Codex 会无差别模仿仓库既有模式，连同次优实现一并照搬，架构漂移速度远超人工开发。

7.3 折中结论

💡 关键认知：真相可能是—— - "Harness"这件事是真的：Agent = Model + Harness，这个抽象成立，数据支撑充分 - "Engineering"里新东西的比例因人而异：对 AI 圈新人是全新范式；对资深软件工程师，很多技巧（关注点分离、观察器-执行器分离、渐进信任）都是老朋友 - 长期风险：基础设施一旦被更强的端到端模型内化，今天苦心经营的 Harness 可能像当年的 CoT prompt 一样变成"自带能力"

对工程师的建议：该做还是做。就算未来模型会把这一层吃掉，现在不做 Harness 的 Agent 就是没法上生产。这不是押注未来，而是解决当下。

八、知识递进总览

回看一圈，整个 Harness Engineering 的逻辑链是这样的：

  ① 模型本身差距收窄（SWE-bench 前沿模型差<1pp）
                    ↓
  ② 应用工程师能动的只剩"模型外面那层"
                    ↓
  ③ 把这一层命名为 Harness（Hashimoto 2026-02-05）
                    ↓
  ④ OpenAI 官方认证 + 实战（2026-02-11，100万行/0手写）
                    ↓
  ⑤ 拆解成七大组件：循环/工具/权限/沙箱/上下文/反馈/Evaluator
                    ↓
  ⑥ 实证数据（Terminal Bench 52.8%→66.5%）证明这层有巨大优化空间
                    ↓
  ⑦ 争议同时出现：是突破还是炒作？
                    ↓
  ⑧ 共识：短期必须做，长期可能被模型内化
